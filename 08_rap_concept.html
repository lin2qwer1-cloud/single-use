1부. 프롤로그 — ABAP 화면 기술 연대기 기초

RAP을 이해하는 가장 빠른 길은 문법이 아니라 역사입니다. ABAP 개발자가 지난 30년간 화면을 만들어온 기술이 왜 계속 바뀌었는지, 각 세대가 무엇을 해결했고 왜 다음 세대에 자리를 내줬는지를 따라가면, RAP이 갑자기 튀어나온 신기술이 아니라 누적된 문제의 최종 답안임이 보입니다.

1.1 제1세대 — Dynpro와 SAP GUI (1990s~)

고전 화면 기술. SE51로 화면을 그리고 PBO(Process Before Output)/PAI(Process After Input) 모듈에 로직을 씁니다. 모듈풀(SE80의 Module Pool)과 리포트의 SELECTION-SCREEN이 모두 여기에 속합니다.

해결한 것버려진 이유
업무 화면을 빠르게 만드는 완결된 도구. 필드-테이블 바인딩, F4, 체크가 프레임워크에 내장.SAP GUI 전용. 웹/모바일 접근 불가, UI가 서버 세션에 강결합(stateful), 화면·로직·데이터가 한 프로그램에 뒤엉킴.

1.2 제1.5세대 — ALV

REUSE_ALV_GRID_DISPLAY → cl_gui_alv_grid → cl_salv_table로 진화한 표 출력의 사실상 표준. 지금도 온프레 현장의 주력입니다. 다만 ALV는 "리포트 출력"에 특화된 것이지 범용 앱 프레임워크가 아니고, 역시 SAP GUI 안에서만 삽니다.

1.3 제2세대 — Web Dynpro ABAP과 FPM (2000s)

"GUI를 벗어나 브라우저로"의 첫 시도. 컴포넌트/뷰/컨트롤러 구조(MVC)를 도입했고, FPM(Floorplan Manager)으로 표준 화면 골격을 재사용했습니다.

해결한 것버려진 이유
브라우저 실행, MVC 분리, 재사용 가능한 컴포넌트.무겁고 느린 렌더링, ABAP 서버에 강결합된 SAP 전용 UI, 모바일 부적합, 화면 자유도 낮음. HTML5 시대가 오자 SAP 스스로 Fiori로 방향 전환.

1.4 전환점 — Fiori와 OData의 등장 (2013~)

SAP는 UI 전략을 SAPUI5(HTML5 라이브러리) + Fiori(디자인 표준)로 전면 교체합니다. 핵심 구조 변화는 이것입니다:

[Dynpro 시대]  화면과 로직이 같은 ABAP 세션 안에 (stateful, GUI 전용)
[Fiori 시대]   브라우저(UI5 앱)  ◀── HTTP/OData ──▶  ABAP 서버 (stateless API)

UI가 서버 밖(브라우저)으로 나가면서, 서버는 화면이 아니라 API(OData 서비스)를 제공하는 역할로 바뀌었습니다. 문제는 그 API를 만드는 일이 고통스러웠다는 것 — 그래서 제3세대가 등장합니다.

1.5 제3세대 — SEGW + SADL + BOPF: "ABAP Programming Model for SAP Fiori"

OData 서비스를 만드는 도구로 SEGW(Gateway Service Builder)가 나왔습니다. 처음엔 엔티티 정의부터 CRUD 메서드(*_GET_ENTITYSET 등)까지 전부 수작업이었습니다. 이후 CDS 뷰를 참조(reference)해 읽기를 자동화하는 SADL, 트랜잭션 로직(생성/수정/삭제, validation, determination, action)을 담는 BOPF가 결합되어 "ABAP Programming Model for SAP Fiori"라는 공식 모델이 됩니다.

해결한 것버려진 이유
CDS 기반 읽기 자동화, BO(Business Object) 개념 도입 — validation/determination/action이라는 어휘는 여기서 확립됨.도구가 조각조각: SEGW(GUI)+CDS(ADT)+BOPF(별도 프레임워크·자체 API·트랜잭션 BOBF)를 오가며 개발. 배우기 어렵고 보일러플레이트 많고 디버깅 힘듦. 클라우드에서 쓸 수 없는 구조.

1.6 제4세대 — RAP (2019~, 현재)

RAP(ABAP RESTful Application Programming Model)은 위 조각들을 언어와 CDS 안으로 흡수했습니다. 2018년 공개, 클라우드(BTP ABAP Environment) 1808에서 최초 릴리스, 온프레는 S/4HANA 1909(ABAP 7.54, 2019)부터 사용 가능합니다. (출처: SAP Community RAP FAQ, SAP RAP Overview 2022)

  • 데이터 모델: CDS 뷰 엔티티 (별도 도구 없음)
  • 트랜잭션 로직: Behavior Definition(BDEF) — BOPF의 BO 개념을 계승하되 전용 언어(BDL)로 선언, 보일러플레이트 제거
  • 서비스 노출: Service Definition/Binding — SEGW 불필요, OData V2/V4 자동 생성
  • UI: Fiori Elements가 CDS 어노테이션을 읽어 화면 자동 생성 — UI5 코딩 없이 List Report/Object Page
  • 도구: 전부 ADT(Eclipse). SAP GUI 트랜잭션 없음
한 줄 요약 — Dynpro는 "GUI 안의 화면", SEGW 시대는 "화면 따로 API 따로 조립", RAP은 "CDS+행동선언만 쓰면 API와 화면이 표준 절차로 따라 나오는 모델"입니다.

2부. RAP의 탄생 배경 기초

2.1 REST와 OData — SM04 세션이 없는 세상

클래식 ABAP 개발자에게 가장 낯선 변화는 stateless입니다. Dynpro에선 사용자가 화면을 띄우는 동안 서버 세션(SM04에 보이는 그것)이 살아있고, 전역 변수·ENQUEUE 잠금이 세션과 함께 유지됩니다. 웹 세상은 다릅니다:

  • REST: 클라이언트(브라우저)가 서버에 요청 하나를 보내고 응답을 받으면 대화는 끝. 서버는 요청 사이에 아무것도 기억하지 않습니다(stateless). 수천 명이 접속해도 세션을 붙들지 않으니 확장에 유리합니다.
  • OData: REST 위에 SAP·Microsoft 등이 표준화한 프로토콜. "엔티티(테이블 비슷한 것)를 URL로 조회/필터/페이징/수정하는 규칙"입니다. 예: GET .../Flight?$filter=CarrierId eq 'AA'&$top=10. Fiori(UI5)는 OData를 말하는 클라이언트입니다.

그럼 stateless인데 "편집 중" 상태는 누가 기억하나? — 이 질문의 RAP 답이 draft(6부)입니다. 잠금은? — ETag 기반 낙관적 잠금(6부)입니다. 클래식의 세션·ENQUEUE가 하던 일이 전부 다른 메커니즘으로 대체된다는 것, 이것이 RAP 학습의 실체입니다.

2.2 S/4HANA 전략과 "Fiori가 기본 UI"

S/4HANA의 표준 신규 화면은 Fiori 앱입니다. SAP GUI 트랜잭션은 유지되지만 신규 기능은 Fiori 우선으로 나옵니다. 고객 개발(CBO)도 같은 흐름을 요구받습니다 — "우리 회사 커스텀 화면도 Fiori로"라는 요구가 오면, UI5를 직접 코딩하는 프런트엔드 개발 없이 ABAP 개발자가 대응할 수 있는 표준 경로가 RAP + Fiori Elements입니다.

2.3 클라우드 전략 — clean core와 ABAP Cloud

  • Clean core: 코어(표준 시스템)를 표준에 최대한 가깝게 유지하고, 확장은 표준이 공개한(released) 안정 인터페이스로만 하자는 SAP의 운영 원칙. 업그레이드 때마다 커스텀이 깨지는 고통(클래식 시대의 user exit/모디피케이션 유산)을 없애는 것이 목적입니다. (출처: SAP clean core 공식)
  • ABAP Cloud: clean core를 강제하는 제한된 ABAP 개발 모델. released API만 호출 가능, Dynpro/SAP GUI 불가, RAP이 유일한 앱 모델. 두 곳에서 동일하게 적용됩니다 — ① BTP ABAP Environment(별칭 Steampunk: SAP가 운영하는 클라우드 ABAP 서버), ② S/4HANA 2022+의 Developer Extensibility(별칭 Embedded Steampunk: 온프레/프라이빗 클라우드 안에서 ABAP Cloud 문법으로 개발).
  • 이 구도에서 RAP은 선택지가 아니라 ABAP Cloud의 유일한 애플리케이션 프레임워크입니다. 클라우드로 갈수록 "RAP을 아느냐"가 "ABAP으로 앱을 만들 수 있느냐"와 동의어가 됩니다.
우리 시스템 위치 — 이 교육 시스템(S/4HANA 2021 온프레, SAP_BASIS 756)은 ABAP Cloud 강제가 없는 "클래식 확장성" 환경입니다. 즉 RAP과 클래식을 모두 쓸 수 있고, RAP 안에서도 비released 오브젝트(SFLIGHT 등)를 참조할 수 있습니다. BTP나 S/4HANA Cloud에선 그게 막힌다는 차이를 10부작 내내 표시합니다.

3부. 업계 현황 (2026) 기초

3.1 시계는 돌아간다 — ECC 유지보수 종료

  • ECC 6.0(EHP6~8)의 mainstream maintenance는 2027년 말 종료. 추가 요금을 내는 extended maintenance가 2030년 말까지 제공되고(EHP0~5는 대상 아님), 이후엔 SAP 관리 프라이빗 클라우드로 한정한 2031~2033 전환 옵션이 발표되어 있습니다. (출처: SEIDOR, Rimini Street, The Register)
  • 전환은 진행 중이지만 완결과는 거리가 멀다: 2025년 조사 기준 59%가 S/4HANA에 전면/부분 라이브(전년 +13%p, Precisely/ASUG). 반면 Gartner 집계로는 2024년 말 기준 ECC 고객의 약 39%만 전환을 마쳤고, 40% 이상이 2027년 이후에도 ECC에 남을 것이라는 전망도 있습니다. (출처: ASUG, CIO, Basis Technologies)

3.2 실무에서 RAP은 얼마나 쓰이나 — 균형 잡힌 현실

솔직한 현실 인식이 중요합니다:

  • 신규 개발의 기본값은 RAP 쪽으로 이동 중: "레거시 제약이 없다면 신규 확장은 RAP이 1순위"가 2026년 현재 일반적 권고입니다. 채용 공고에서도 RAP/CDS/Fiori가 필수 요건화되는 추세입니다. (출처: SAP Press)
  • 그러나 현장은 혼용: 위 통계처럼 ECC가 여전히 대량으로 살아있고, S/4HANA 온프레 현장도 기존 자산(리포트·ALV·BAPI·인터페이스)이 클래식입니다. RAP은 클래식의 "대체"가 아니라 병행 — 서비스가 필요 없는 배치 잡, 소형 유틸, 레거시 유지보수는 계속 클래식입니다. "RAP만 알면 된다"도 "클래식만으로 버틴다"도 틀린 명제입니다.
  • RAP의 적소: 화면(Fiori)이 필요한 트랜잭션 업무 앱, 외부에 노출할 API, 클라우드 이행을 염두에 둔 신규 개발.

3.3 개발자 커리어 관점

  • RAP/CDS/BTP 역량에 보수 프리미엄이 붙는다는 보고가 이어집니다 — 시장 조사·구인 플랫폼 기준 미국 ABAP 개발자 평균 연봉은 약 $142K(Glassdoor), RAP+CDS+BTP 보유자는 동일 연차 클래식 개발자 대비 크게 높은 보수를 받는다는 벤더 보고가 있습니다(정확한 배율은 조사마다 상이). (출처: Glassdoor, ZipRecruiter)
  • 도구 측면의 새 흐름: SAP Joule for Developers가 ADT에 GA되어 RAP BO 생성·테스트 생성을 지원하고, VS Code용 ABAP Cloud 확장이 예고되는 등 RAP이 SAP의 AI·신도구 투자가 집중되는 트랙입니다. (출처: SAP Community RAP 토픽)
  • 결론: 클래식 ABAP 경력자에게 RAP은 "새 언어"가 아니라 기존 ABAP+CDS 위의 새 모델입니다. 학습 곡선의 대부분은 문법이 아니라 사고방식(stateless, 선언적, 프레임워크 주도) 전환입니다. 이 문서가 그 전환을 돕습니다.

4부. 큰 그림 — RAP 아키텍처 계층도 기초

4.1 전체 스택 한 장

RAP 개발은 아래 계층을 아래에서 위로 쌓는 일입니다. 각 계층은 독립된 오브젝트(괄호는 09 실습의 실물 이름)입니다.

┌────────────────────────────────────────────────────────────┐
│  Fiori Elements 앱 (List Report / Object Page)             │  ← @UI 어노테이션이 화면을 결정
├────────────────────────────────────────────────────────────┤
│  OData 서비스 (V2/V4)                                      │  ← 자동 생성됨
├────────────────────────────────────────────────────────────┤
│  Service Binding (SRVB)  YUI_FLIGHT_O4                     │  ← 프로토콜/버전 선택 + 게시(Publish)
│  Service Definition (SRVD)  YUI_FLIGHT_O4                  │  ← 무엇을 노출할지 목록
├────────────────────────────────────────────────────────────┤
│  Projection(소비) 계층                                     │
│    CDS Projection View  YC_FLIGHT   (+@UI 어노테이션)      │  ← 특정 용도(UI)용으로 필드 선별
│    Projection BDEF      YC_FLIGHT   (use create; ...)      │  ← 노출할 행동 선별
├────────────────────────────────────────────────────────────┤
│  Business Object(BO) 계층                                  │
│    Behavior Definition  YR_FLIGHT (BDEF: managed+draft)    │  ← 행동 선언 (무엇을 할 수 있나)
│    Behavior Pool        YCL_BP_R_FLIGHT (lhc_* 핸들러)     │  ← 행동 구현 (validation 등)
├────────────────────────────────────────────────────────────┤
│  데이터 모델 계층                                          │
│    CDS Root View Entity YR_FLIGHT                          │  ← 의미 부여·계산필드·association
├────────────────────────────────────────────────────────────┤
│  DB 테이블  YSFLIGHT (+ draft 테이블 YSFLIGHT_D)           │  ← 저장소
└────────────────────────────────────────────────────────────┘

외우는 순서: 테이블 → CDS(뷰) → BDEF(행동) → 핸들러 클래스 → Projection(뷰+행동) → SRVD → SRVB → (자동)OData → (자동)Fiori. 09 문서의 Step 1~7이 정확히 이 순서입니다. 09 실습 ▸

4.2 클래식 개발과의 1:1 대응표

클래식으로 같은 앱(항공편 관리 화면)을 만든다면 무엇에 해당하는가 — 이 표가 이 문서에서 가장 실용적인 표입니다.

4.3 "RAP BO"라는 사고방식

클래식에선 프로그램이 주인공이고 데이터는 재료였습니다. RAP에선 Business Object(BO)가 주인공입니다. BO = 구조(CDS 엔티티 트리) + 행동(BDEF) + 구현(핸들러)의 묶음이고, UI든 API든 EML이든 모두 이 BO를 통해서만 데이터를 만집니다. "화면을 만든다"가 아니라 "BO를 만들면 화면은 따라온다"가 RAP의 사고방식입니다.

6부. 트랜잭션 모델 개념 기초심화

6.1 managed — 프레임워크가 CRUD를 대신한다 기초

BDEF 첫 줄이 managed;면, create/update/delete의 표준 동작(버퍼 관리, 테이블 저장)은 프레임워크가 구현합니다. 개발자는 업무 로직(validation, determination, action)만 씁니다. SM30 유지보수 화면을 떠올리면 됩니다 — 저장 로직을 직접 짠 적 없듯이, managed에선 INSERT/UPDATE 문을 쓰지 않습니다. 신규 테이블 기반 개발의 기본값입니다.

실물 예 — 09 실습의 BDEF 선언부 (YR_FLIGHT, 발췌):

managed implementation in class ycl_bp_r_flight unique;
strict;
with draft;

define behavior for YR_FLIGHT alias Flight
persistent table ysflight
draft table ysflight_d
etag master LocalLastChangedAt
lock master total etag LastChangedAt
{
  create; update; delete;
  field ( readonly ) OccupancyRate, CreatedBy, ...;
  validation validateCarrier on save { create; field CarrierId; }
  ...
}

선언 몇 줄이 클래식에서 프로그램 하나 분량이던 것(화면 잠금, 버퍼, 저장, 상태 관리)을 대체합니다. 전체 해부는 09 Step 3 ▸

6.2 unmanaged — 레거시 로직을 재활용할 때 기초

unmanaged;면 create/update/delete/read/lock을 전부 개발자가 구현합니다. 언제 쓰나? — 이미 검증된 저장 로직이 있을 때입니다. 수년간 쓰인 BAPI·함수 모듈·클래스가 있는데 화면만 Fiori로 바꾸고 싶다면, 그 로직을 버리고 managed로 다시 만들 이유가 없습니다. unmanaged BO가 기존 API를 감싸는(wrapping) 어댑터가 됩니다. 완결 지면 예제는 10 §3 ▸

중간 형태도 있습니다: managed with unmanaged save(버퍼는 프레임워크, 최종 저장만 직접 — 예: 저장을 BAPI로), managed with additional save(표준 저장 + 추가 작업 — 예: 변경이력 기록). 선택 기준 결정 트리는 10 §3에.

6.3 draft — stateless 세상의 "편집 중" 기초 심화

2.1의 질문으로 돌아갑니다. 브라우저 요청 사이에 서버가 아무것도 기억하지 않는다면, 사용자가 30분째 입력 중인 반쯤 완성된 데이터는 어디에? — draft 테이블(실물: YSFLIGHT_D)입니다. with draft; 한 줄이면:

  • 자동 중간저장: 입력할 때마다 draft 행이 갱신됨. 브라우저가 꺼져도 이어서 작업(Resume) 가능
  • 잠금 대체: draft 행의 소유자가 편집권을 가짐(exclusive lock) — ENQUEUE 없이 "누가 편집 중" 관리
  • 표준 액션 자동 제공: Edit(활성→draft 복사), Activate(draft→활성 저장), Discard, Resume — BDEF에 draft action으로 선언만 하면 됨

심화 비용도 있습니다: draft 테이블이 하나 더 생기고(구조는 활성 테이블 + %admin include), validation이 draft 저장 시점(Prepare)에도 돌며, 조회가 "활성+내 draft" 합성이라 디버깅이 한 겹 복잡해집니다. 단순 조회 앱이면 draft 없이 가는 것도 선택입니다.

6.4 잠금 — ENQUEUE에서 ETag로 심화

클래식: 편집 시작 시 ENQUEUE(비관적 잠금) — 남이 못 들어오게 문을 잠급니다. 세션이 살아있어야 유지됩니다.
RAP: 낙관적 잠금 — 문을 잠그지 않고, 행마다 버전 도장(etag master LocalLastChangedAt)을 찍습니다. 저장하러 왔을 때 도장이 내가 읽었을 때와 다르면 "그 사이 누가 바꿨음" → 저장 거부. lock master total etag LastChangedAt은 BO 트리 전체의 변경 감지 기준입니다. 단, 수정 요청 처리 중에는 RAP도 짧게 비관적 잠금을 걸어 동시 쓰기를 막습니다(draft의 편집권은 별도 메커니즘).

6.5 RAP LUW — UPDATE TASK와 무엇이 다른가 심화

클래식 SAP LUWRAP LUW
프로그램이 아무 데서나 DB를 갱신할 수 있고, 지연 갱신은 CALL FUNCTION ... IN UPDATE TASK로 등록 후 COMMIT WORK가 실행2단계가 강제됨: interaction phase(create/update/validation… — 상태는 트랜잭션 버퍼에만) → save sequence(finalize → check_before_save → save — 여기서만 DB 기록)
커밋 시점·순서를 개발자가 관리 (잊으면 부분 커밋 사고)COMMIT ENTITIES(또는 UI가 대신)가 save sequence 전체를 실행 — 검증 실패 시 전체가 깨끗하게 무산
번호 채번을 커밋 직전 수작업 배치late numbering이 save 단계에서 표준으로 처리

핵심: interaction phase에서는 DB에 직접 쓰지 않는다. managed에서 개발자 코드(validation 등)가 DB에 INSERT하는 것은 모델 위반입니다. 부수 기록(로그 등)은 additional save에서.

7부. EML 개관 심화

EML(Entity Manipulation Language)은 ABAP 코드에서 RAP BO를 다루는 구문입니다. 왜 필요한가 — RAP BO는 단순 테이블이 아니라 버퍼·draft·권한·validation을 거치는 살아있는 오브젝트이기 때문입니다. SELECT ... FROM ysflight은 저장된 행만 볼 뿐 draft·계산필드·행동을 모릅니다. BO로서 접근하려면 EML이 유일한 문입니다.

" 읽기 — BO를 통해 (계산필드 OccupancyRate 포함, 키는 CDS 필드명)
READ ENTITIES OF yr_flight
  ENTITY Flight
  ALL FIELDS WITH VALUE #( ( CarrierId = 'AA' ConnectionId = '0017'
                              FlightDate = '20230622' ) )
  RESULT DATA(lt_result)
  FAILED DATA(ls_failed).

" 쓰기 — validation을 통과해야 반영됨
MODIFY ENTITIES OF yr_flight
  ENTITY Flight
  UPDATE FIELDS ( Price ) WITH VALUE #( ( CarrierId = 'AA' ... Price = '500.00' ) )
  FAILED DATA(ls_f) REPORTED DATA(ls_r).
COMMIT ENTITIES.   " ← save sequence 실행 (6.5)

용도: ① BO 핸들러 안에서 자기/남의 BO 접근(IN LOCAL MODE — 권한·feature control 생략), ② 배치/인터페이스 코드가 BO를 통해 데이터 조작(validation 재활용), ③ 테스트. 라이브 시스템 실측 예제와 결과는 09 §3 EML 실습 ▸
