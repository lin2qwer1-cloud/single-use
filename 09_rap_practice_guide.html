§1 실습 개요 기초

만드는 것: 항공편 관리 Fiori 앱. List Report에서 항공편을 검색·조회하고, Object Page에서 상세를 보고, Edit로 수정(draft 자동 저장)하고, 저장 시 4종 validation이 데이터를 지켜줍니다. UI5 코드는 한 줄도 쓰지 않습니다.

스택 전체 그림은 08 4부 아키텍처 ▸ 를 옆에 두고 보세요. 이 문서의 Step 번호가 그 계층도와 1:1입니다.

Step 1. 테이블 — YSFLIGHT & YSFLIGHT_D 기초

① 이 단계에서 만드는 것

저장 테이블. SFLIGHT의 구조를 복사한 YSFLIGHT에 RAP 관리필드 5종을 추가했습니다. 그리고 draft용 YSFLIGHT_D가 별도로 필요합니다.

③ 해설 — 관리필드 5종은 왜 필요한가

필드도메인쓰임
created_by/created_atabp_creation_*생성자/생성시각 — 프레임워크가 자동 기록 (CDS의 @Semantics.user.createdBy 등과 짝)
last_changed_by/last_changed_atabp_lastchange_*최종 변경자/시각 — lock master의 total etag로 사용 (BO 전체 충돌 감지)
local_last_changed_atabp_locinst_lastchange_tstmpl이 엔티티 인스턴스의 변경 시각 — etag master로 사용 (행 단위 낙관적 잠금)

즉 마지막 두 필드는 "있으면 좋은 감사 필드"가 아니라 잠금 메커니즘의 부품입니다(08 6.4 ▸). 신규 RAP 테이블을 설계할 땐 이 5종을 표준 세트로 넣는 것이 관례입니다.

주목할 차이 세 가지: ① 필드명이 테이블 필드명(carrid)이 아니라 CDS 엔티티 필드명 소문자(carrierid) — draft는 BO 관점의 저장소이기 때문. ② 계산필드 occupancyrate도 컬럼으로 존재 — 편집 중 값을 들고 있어야 하므로. ③ 끝의 "%admin" include sych_bdl_draft_admin_inc — draft 소유자·시각 등 관리 정보. ADT에서 BDEF의 draft table 이름에 커서를 두고 퀵픽스(Ctrl+1)를 쓰면 이 테이블을 자동 생성해 줍니다 — 수작업으로 만들지 않는 것이 원칙입니다.

클래식이라면 — 테이블 만들기는 같습니다(SE11/ADT). 다른 점은 "편집 중 상태"를 위해 메모리(전역 변수)와 ENQUEUE에 의존했지, 별도 테이블을 두지 않았다는 것. draft 테이블은 그 상태를 DB로 내린 것입니다.

⑤ 확인 방법

ADT에서 두 테이블을 열고 활성(no ⚠) 상태 확인. SE16/데이터 프리뷰(F8)로 YSFLIGHT에 데이터가 있는지 확인(이 시스템엔 SFLIGHT 사본 데이터가 로드되어 있음).

Step 2. Interface View — YR_FLIGHT + Value Help 3종 기초

① 이 단계에서 만드는 것

BO의 구조가 될 root view entity. 테이블 필드에 CamelCase 별칭을 붙이고, association으로 관련 정보를 연결하고, 계산필드(점유율)를 추가합니다.

② 소스 전문 — YR_FLIGHT

@AccessControl.authorizationCheck: #NOT_REQUIRED
@EndUserText.label: 'Flight Root View (RAP GRID)'
define root view entity YR_FLIGHT
  as select from ysflight
  association [0..1] to scarr      as _Carrier    on $projection.CarrierId = _Carrier.carrid
  association [0..1] to spfli      as _Connection on  $projection.CarrierId    = _Connection.carrid
                                                  and $projection.ConnectionId = _Connection.connid
  association [0..1] to saplane    as _Plane      on $projection.PlaneType = _Plane.planetype
  association [0..1] to I_Currency as _Currency   on $projection.CurrencyCode = _Currency.Currency
{
  key carrid                as CarrierId,
  key connid                as ConnectionId,
  key fldate                as FlightDate,
      @Semantics.amount.currencyCode: 'CurrencyCode'
      price                 as Price,
      currency              as CurrencyCode,
      planetype             as PlaneType,
      seatsmax              as MaximumSeats,
      seatsocc              as OccupiedSeats,
      cast( case when seatsmax > 0
                 then division( seatsocc * 100, seatsmax, 1 )
                 else cast( 0 as abap.dec(5,1) )
            end as abap.dec(5,1) ) as OccupancyRate,
      @Semantics.amount.currencyCode: 'CurrencyCode'
      paymentsum            as PaymentsSum,
      seatsmax_b            as MaximumSeatsBusiness,
      seatsocc_b            as OccupiedSeatsBusiness,
      seatsmax_f            as MaximumSeatsFirst,
      seatsocc_f            as OccupiedSeatsFirst,
      @Semantics.user.createdBy: true
      created_by            as CreatedBy,
      @Semantics.systemDateTime.createdAt: true
      created_at            as CreatedAt,
      @Semantics.user.lastChangedBy: true
      last_changed_by       as LastChangedBy,
      @Semantics.systemDateTime.lastChangedAt: true
      last_changed_at       as LastChangedAt,
      @Semantics.systemDateTime.localInstanceLastChangedAt: true
      local_last_changed_at as LocalLastChangedAt,
      _Carrier,
      _Connection,
      _Plane,
      _Currency
}

③ 해설

  • define root view entity — 이 뷰가 BO 트리의 뿌리임을 선언. 단일 엔티티 BO라도 root는 필수입니다. (자식이 있으면 자식 쪽에 composition을 씁니다 — 이 실습은 단일 엔티티)
  • CamelCase 별칭 — RAP 세계의 필드명은 CDS 별칭입니다. 이후 BDEF·핸들러·EML·OData·화면까지 전부 CarrierId로 통하고, carrid는 저장 시점에만 등장합니다(BDEF의 mapping).
  • OccupancyRate — cast(case...division(...)) 계산필드. DB에 저장되지 않고 조회 시 계산됩니다. 뒤에서 BDEF가 field ( readonly )로 막고, draft 테이블에는 컬럼으로 존재하는 것까지가 한 세트입니다.
  • @Semantics.user.createdBy 등 — 관리필드의 의미를 선언. managed 프레임워크가 이 어노테이션을 보고 자동으로 채웁니다.
  • association 4개는 화면·후속 뷰에서 쓸 연결 통로입니다. 문법은 01_cds_concept.html 3부 참조.

Step 3. Behavior Definition — BDEF 해부 기초심화

① 이 단계에서 만드는 것

BO의 행동 선언. ADT에서 YR_FLIGHT(DDLS) 우클릭 → New Behavior Definition을 하면 뼈대가 생성되고, 구현 유형(managed)을 고릅니다. BDEF는 CDS와 같은 이름을 씁니다.

② 소스 전문 — BDEF YR_FLIGHT

managed implementation in class ycl_bp_r_flight unique;
strict;
with draft;

define behavior for YR_FLIGHT alias Flight
persistent table ysflight
draft table ysflight_d
etag master LocalLastChangedAt
lock master total etag LastChangedAt
authorization master ( global )
{
  create;
  update;
  delete;

  field ( readonly ) OccupancyRate, CreatedBy, CreatedAt, LastChangedBy, LastChangedAt, LocalLastChangedAt;
  field ( readonly : update ) CarrierId, ConnectionId, FlightDate;
  field ( mandatory : create ) CarrierId, ConnectionId, FlightDate;

  validation validateCarrier    on save { create; field CarrierId; }
  validation validateConnection on save { create; field CarrierId, ConnectionId; }
  validation validatePlaneType  on save { create; field PlaneType; }
  validation validateSeats      on save { create; field MaximumSeats, OccupiedSeats,
                                          MaximumSeatsBusiness, OccupiedSeatsBusiness,
                                          MaximumSeatsFirst, OccupiedSeatsFirst; }

  draft action Edit;
  draft action Activate;
  draft action Discard;
  draft action Resume;
  draft determine action Prepare
  {
    validation validateCarrier;
    validation validateConnection;
    validation validatePlaneType;
    validation validateSeats;
  }

  mapping for ysflight
  {
    CarrierId             = carrid;
    ConnectionId          = connid;
    FlightDate            = fldate;
    Price                 = price;
    CurrencyCode          = currency;
    PlaneType             = planetype;
    MaximumSeats          = seatsmax;
    OccupiedSeats         = seatsocc;
    PaymentsSum           = paymentsum;
    MaximumSeatsBusiness  = seatsmax_b;
    OccupiedSeatsBusiness = seatsocc_b;
    MaximumSeatsFirst     = seatsmax_f;
    OccupiedSeatsFirst    = seatsocc_f;
    CreatedBy             = created_by;
    CreatedAt             = created_at;
    LastChangedBy         = last_changed_by;
    LastChangedAt         = last_changed_at;
    LocalLastChangedAt    = local_last_changed_at;
  }
}

⑤ 확인 방법 / ⑥ 함정

ADT에서 BDEF 활성화 시 문제 뷰에 경고가 없어야 합니다. 이 시점에 "구현 클래스에 validation 메서드가 없다"는 경고가 뜨는 게 정상 — Step 4에서 채웁니다(BDEF에서 Ctrl+1 퀵픽스로 메서드 뼈대 자동 생성 가능).

함정(실측) — 온라인 예제의 strict ( 2 )를 이 시스템에 붙여넣으면 "Unexpected character" 구문오류. 756은 숫자형 strict를 모릅니다. strict;로 바꾸면 됩니다.

Step 4. Behavior Pool — lhc_ 핸들러 심화

③ 해설 — RAP 핸들러의 관용구 5가지

  1. FOR VALIDATE ON SAVE ... IMPORTING keys FOR Flight~validateCarrier — 메서드 시그니처가 BDEF 선언과 이름으로 연결됩니다. keys는 "검증 대상 인스턴스들의 키 목록" — 항상 대량(테이블) 처리가 기본입니다. 한 건씩 호출되지 않습니다.
  2. READ ENTITIES ... IN LOCAL MODE — 검증할 값은 파라미터로 오지 않습니다. 키만 받고, 현재 값(트랜잭션 버퍼+draft 포함)은 EML로 직접 읽습니다. IN LOCAL MODE는 "BO 내부 접근이니 권한·feature control 생략" 표시입니다.
  3. %tky — transactional key. 키 + draft 여부(%is_draft)를 묶은 구조로, 프레임워크와 대화할 때의 인스턴스 신분증입니다.
  4. failed / reported — 반환값 대신 쓰는 암묵 파라미터. failed-flight에 넣으면 "이 인스턴스 저장 거부", reported-flight의 %msg가 화면 메시지, %element-carrierid는 "이 필드에 빨간 표시" 지정입니다. 클래식의 MESSAGE E000 ... FIELD carrid에 해당.
  5. %state_area — 메시지 수명 관리 구역. 검증 시작 시 자기 구역을 빈 값으로 APPEND(=이전 메시지 지우기)하고, 오류 시 같은 구역으로 메시지를 넣습니다. draft 재검증 때 낡은 메시지가 화면에 쌓이는 것을 막는 표준 관용구입니다.

Step 7. Fiori Elements 프리뷰 기초

① 이 단계에서 하는 것

코딩 없이 화면 확인. SRVB에서 엔티티(Flight) 선택 → Preview — 브라우저에 List Report가 뜹니다.

③ 확인 시나리오 (화면 흐름)

  1. List Report: 필터바에 CarrierId/ConnectionId/FlightDate(= @UI.selectionField), Go를 누르면 표(컬럼 = @UI.lineItem 순서). CarrierId에서 F4 → 항공사 목록(YI_CARRIER_VH). 항공사를 고른 뒤 ConnectionId F4 → 그 항공사의 노선만 나오는지(additionalBinding 효과) 확인.
  2. Object Page: 행 클릭 → 상세 화면. 상단 제목이 CarrierId/FlightDate(= headerInfo), "Flight Details" 구역(= facet)에 identification 필드들.
  3. Edit와 draft: Edit 클릭 → 편집 모드. 키 3종이 잠겨 있고(readonly : update) 다른 필드는 입력 가능. 값을 바꾸고 저장하지 않은 채 브라우저 탭을 닫았다가 다시 열면 — "편집 중이던 draft가 있다"며 이어하기(Resume)를 제안합니다. 이것이 draft의 체감입니다.
  4. validation 체감: OccupiedSeats에 MaximumSeats보다 큰 값 입력 후 저장 → 저장이 거부되고 해당 필드에 "Occupied seats (…) exceed maximum seats (…)" 오류가 필드 단위로 표시됩니다(= %element-occupiedseats의 효과).
  5. Create: 신규 생성 화면에서 키 3종에 필수(*) 표시(mandatory : create), 존재하지 않는 항공사 코드 입력 시 validateCarrier 오류.
여기서 멈추고 음미할 것 — 우리는 UI5 코드를 한 줄도 안 썼습니다. 화면의 모든 요소가 Step 3(BDEF)와 Step 5(@UI)의 선언에서 나왔습니다. 화면을 바꾸고 싶으면 코드가 아니라 선언을 바꿉니다.

§3 EML 실습 — 라이브 실측 심화

개념은 08 7부 ▸. 여기선 이 시스템에서 실제로 실행하고 출력을 캡처한 코드입니다(2026-07-20). 점유율 있는 항공편 2건을 BO를 통해 읽습니다:

DATA lv_msg TYPE string.
SELECT carrid, connid, fldate FROM ysflight
  WHERE seatsocc > 0
  ORDER BY carrid, connid, fldate
  INTO TABLE @DATA(lt_keys)
  UP TO 2 ROWS.
READ ENTITIES OF yr_flight
  ENTITY Flight
  ALL FIELDS WITH VALUE #(
    FOR ls_k IN lt_keys
    ( CarrierId = ls_k-carrid
      ConnectionId = ls_k-connid
      FlightDate = ls_k-fldate ) )
  RESULT DATA(lt_result)
  FAILED DATA(ls_failed).
lv_msg = |read={ lines( lt_result ) }|.
LOOP AT lt_result INTO DATA(ls_r).
  lv_msg = lv_msg &&
    | \| { ls_r-CarrierId }/{ ls_r-ConnectionId }/{ ls_r-FlightDate }| &
    | occ={ ls_r-OccupancyRate }%| &
    | seats={ ls_r-OccupiedSeats }/{ ls_r-MaximumSeats }| &
    | price={ ls_r-Price } { ls_r-CurrencyCode }|.
ENDLOOP.

실제 출력(원문):

read=2 | AA/0017/20230622 occ=96.6% seats=372/385 price=423.44 USD | AA/0017/20230624 occ=95.5% seats=315/330 price=423.44 USD

읽는 법:

  • 키를 carrid로 SELECT했지만 EML에는 CarrierId로 넣습니다 — BO 세계의 이름은 CDS 별칭(Step 2).
  • OccupancyRate가 결과에 실려 있습니다(372/385=96.6% — 검산 일치). SELECT * FROM ysflight에는 이 필드가 없습니다. BO를 통해 읽었기에 계산필드가 함께 옵니다.
  • 같은 코드를 데이터 없는 키로 돌리면 read=0일 뿐 오류가 아닙니다(첫 실측: keys=3 read=3 failed=0 ... occ=0.0% seats=0/128 — 좌석 0인 행은 점유율 0.0%로 계산됨).
실행 환경 노트 — 이 실측은 콘솔 러너 환경에서 수행했습니다. ADT가 있다면 같은 코드를 if_oo_adt_classrun 클래스의 main에 넣고 out->write( lv_msg )로 F9 실행하는 것이 표준입니다. 쓰기(MODIFY+COMMIT ENTITIES)까지 실습하려면 §4 심화 과제를 마친 뒤 테스트 데이터로 시도해 보세요 — validation이 지켜주니 안심하고 실험해도 됩니다.

정답 보기

② lhc_flight — 정의부에 메서드 선언 추가:

    METHODS validatedate FOR VALIDATE ON SAVE
      IMPORTING keys FOR Flight~validateDate.

구현부 (관용구는 Step 4의 5가지 그대로):

  METHOD validatedate.
    READ ENTITIES OF yr_flight IN LOCAL MODE
      ENTITY Flight
        FIELDS ( FlightDate ) WITH CORRESPONDING #( keys )
      RESULT DATA(lt_flight).

    LOOP AT lt_flight INTO DATA(ls_flight).
      APPEND VALUE #( %tky        = ls_flight-%tky
                      %state_area = 'VALIDATE_DATE' ) TO reported-flight.

      IF ls_flight-FlightDate >= cl_abap_context_info=>get_system_date( ).
        CONTINUE.
      ENDIF.
      APPEND VALUE #( %tky = ls_flight-%tky ) TO failed-flight.
      APPEND VALUE #( %tky        = ls_flight-%tky
                      %state_area = 'VALIDATE_DATE'
                      %msg        = new_message_with_text(
                          severity = if_abap_behv_message=>severity-error
                          text     = |Flight date { ls_flight-FlightDate DATE = USER } is in the past.| )
                      %element-flightdate = if_abap_behv=>mk-on ) TO reported-flight.
    ENDLOOP.
  ENDMETHOD.

검증: 프리뷰에서 Create → 과거 날짜 입력 → 저장 시 FlightDate 필드에 오류가 떠야 합니다. (참고: cl_abap_context_info=>get_system_date( )는 ABAP Cloud 호환 날짜 조회 — sy-datum도 이 시스템에선 동작하지만 클라우드 이행을 생각하면 전자가 습관으로 좋습니다.)
