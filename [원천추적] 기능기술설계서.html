A. 기능설계

A1. 개요 · 배경

CCA 배부 사이클(어세스먼트·배분·간접활동배분)이 다단으로 실행되면 최종 리시버에 집계된 금액의 발생 원천을 표준 기능으로 식별할 수 없다. 본 기능은 한 회계월의 배부 전기를 실행 순서대로 재생하여 모든 최종 잔액·유출액을 (원천 오브젝트·원가요소) × (최종 오브젝트·원가요소) 전수 조합으로 분해해 제공한다.

B4. 처리 로직 상세

B4.1 전체 흐름

 ① WRTTP 정규화: '04' 외 전부 → '01' (실적이면 VERSN '000' 고정)
 ② 엣지 추출     : 실적 get_actual_edges / 계획 get_plan_edges
 ③ 자기비용 수집 : get_self_costs — 사이클 VRGNG "제외" 전 라인 (②와 정확히 상보)
 ④ 아웃플로우    : KSPA/KSOS/KOAO/KOAL 음수 라인 → #COPA/#SETTLEMENT행 rule B 의사엣지 (부호 반전)
 ⑤ 전파          : propagate_origins (B4.3)
 ⑥ 컨텍스트 스탬프: kokrs/gjahr/versn/wrttp를 전 행에 기록

B4.2 엣지 분류 규칙

rule판정 (derive_rule)대상 VRGNG전파 방식
UPAROB 공백 제거 후 길이 ≤ 2(모든)전파 제외 — 리시버 앵커 UNTRACEABLE state로 직등록
BVRGNG 4번째 글자 = 'U'RKIU, RKPU (+아웃플로우 강제 B)풀 = sender 전체 state — 가중평균 이동
A그 외RKIV, RKPV, RKIB풀 = sender의 같은 원가요소 state — 계정보존 이동

B4.3 전파 알고리즘 (propagate_origins)

 초기화: state ← 자기비용(origin=자기) ∪ rule-U 엣지(UNTRACEABLE)
         ev_in ← Σ state

 FOR exec_seq (오름차순):            ← 실제 전기 시각 순서로 재생
   FOR rule IN (A, B):               ← 같은 시각이면 A(계정보존) 먼저
     pool[sender(,kstar)] ← Σ state  ; sent ← Σ edge
     FOR each edge:
       IF pool = 0:                  ← 분모 0 폴백
         state(receiver) += UNTRACEABLE(origin=SENDER, kstar=edge.kstar)   ┐
         ev_in += edge.amount                                              │ v2.0 수정:
       ELSE:                         ← 비례 이동                            │ origin은 반드시
         이동조각.amount = state.amount × edge.amount ÷ pool               │ sender (자기참조
         이동조각.cur_kstar = edge.kstar ; origin_* 불변                   │ 터미널 금지)
     sender state ×= (pool − sent) ÷ pool                                  ┘
   압축: 6키(obj,cur_kstar,origin 4종) HASHED 수동 집계  ← EMPTY KEY 테이블은 COLLECT 불가(실측)
   프루닝: |amount| < ε(0.01) → origin_type='PRUNED'로 출력 방출 (소멸 금지 — 대사 보존)

 마감: 6자리 → 2자리 반올림, 잔차는 final_objnr 버킷별로 그 버킷 최대금액 행에 흡수
       (v2.0: 전역 1회 보정 → 버킷별 보정. 터미널 버킷 센트 일치 보장)
 ev_out ← Σ et_trace.amount     불변식: ev_in = ev_out
  • 정밀도 정책 — 내부 P16.6, 출력 P12.2. 반올림은 마지막 1회 + 버킷별 잔차보정.
  • 불변식 근거 — 이동+잔류=원금(비례 항등식), 프루닝·폴백 모두 방출/기록되므로 유입 총량은 반드시 출력 어딘가에 존재.

B5. 예외 · 성능 · 제약

구분내용
성능전 SELECT가 단일 KOKRS·단일 월 범위로 제한 — 대상 데이터 자체가 작음
모든 집계는 HASHED 테이블 O(1) 수동 COLLECT; 전파는 exec_seq 묶음 단위로만 순회
조각 폭발은 라운드별 6키 압축 + ε-프루닝으로 억제
제약단일 월 단위 (기간 범위 확장은 시그니처 변경 필요)
YYYYMM→GJAHR/POPER 변환은 calendar=fiscal 가정 (변형 회계연도 시 T009 변환 추가 필요)
전파는 KOKRS 전역 고정 — BUKRS/GSBER로 전파를 제한하면 대사 불성립 (표시 필터로만 제공)
UNTRACEABLE 비중이 크면 원천 데이터 품질(PAROB 누락·시각 역전) 점검 신호로 해석

B9. 테스트 명세 · 결과 (2026-07-23 라이브 실측)

B9.1 테스트 방법

  • 헤드리스 실행 — SUBMIT ycca_trace_origin WITH p_noalv='X' ... AND RETURN 후 IMPORT ... FROM MEMORY ID 'YCCA_TRACE_TEST'로 rows/in/out/sum 검증.
  • 이중 검증 원칙 — 대사(diff=0)만으로는 분류 결함을 못 잡는다(실측: 자기참조 터미널 결함이 대사 통과 상태로 잠복). 반드시 상위 금액 행 판독(터미널 자기참조 0건, origin의 업무적 타당성)을 병행한다.
  • 독립 대사 — 최종 오브젝트별 엔진 귀속합계 vs 명세서 직접 합계(SUM wkgbtr) 비교.
