1부. 수불부란 무엇인가 업무

1.1 정의

수불부(재고수불장, Stock Ledger)는 일정 기간 동안 자재별 재고가 "얼마에서 시작해서(기초), 무엇이 들어오고(입고), 무엇이 나가고(출고), 어떤 조정을 거쳐(조정), 얼마로 끝났는가(기말)"를 수량과 금액 양쪽으로 한 줄에 보여주는 장부입니다.

  기초(Opening)  +  입고(Inflow)  −  출고(Outflow)  ±  조정(Adjustment)  =  기말(Closing)

  opn_qty/amt      in_*_qty/amt     out_*_qty/amt      adj_*_amt            cls_qty/amt
                                                                              ↑
                                            diff = 좌변 − 우변  (0이어야 정상) ┘

1.2 왜 필요한가

  • 결산 검증 — 월말/기말 재고금액(재무제표의 재고자산)이 이동 내역과 수학적으로 맞는지 확인
  • 감사 대응 — "이 자재 재고가 왜 이 금액인가?"에 대해 기초부터 기말까지의 흐름으로 답변
  • 이상 감지 — diff ≠ 0이면 분류 누락·데이터 오류·기간 경계 문제 중 하나가 존재
diff=0의 의미와 한계 — diff=0은 "질량 보존"(모든 이동이 어딘가로 분류됨)을 뜻할 뿐, 분류의 품질(구매입고가 진짜 IN_PO로 갔는지)까지 보장하지는 않습니다. 분류 품질은 6부의 폴백(OTH) 모니터링과 샘플 추적(5부)으로 확인합니다.

2부. 원천 테이블 & ERD

2.1 ERD — 조인 관계 한눈에 개발

중앙의 CKMLHD가 허브입니다 — 자재원장은 자재를 matnr이 아니라 원가추정번호 kalnr로 식별하므로, 잔액(FCML_REP_V)과 이동(MLDOC)을 자재와 연결하려면 반드시 CKMLHD를 거칩니다. MLDOC은 원천 자재문서(MATDOC)를 awref/aworg/awitem으로 참조하고, 매핑 테이블(YTCO_SLMAP)은 조인이 아니라 규칙 매칭(=/와일드카드)으로 연결됩니다. 1*: matdoc은 논리적으로 1건이지만 물리적으로 여러 행일 수 있습니다(아래 함정 참조).

4부. SQLScript 한줄 해부 ★ 개발

메서드 선언부터 봅니다. 이후 10개 블록을 실행 순서대로 해부합니다. (소스는 2026-07-13 수정 반영 현행본)

METHOD get_stock_ledger BY DATABASE FUNCTION
                        FOR HDB LANGUAGE SQLSCRIPT
                        OPTIONS READ-ONLY
                        USING fcml_rep_v ckmlhd mara makt mldoc matdoc ytco_slmap.
  • BY DATABASE FUNCTION + CDS 쪽 FOR TABLE FUNCTION — 프로시저가 아닌 함수형 AMDP. 결과를 RETURN SELECT로 돌려줍니다 (03 문서 2.5).
  • OPTIONS READ-ONLY — 조회 전용 보장(품질 체크리스트 ④).
  • USING — 참조하는 7개 오브젝트만 정확히 나열(체크리스트 ⑤).

1lt_master — kalnr ↔ 자재 다리

lt_master =
  SELECT hd.mandt, hd.kalnr, hd.matnr, hd.bwkey, hd.bwtar,
         makt.maktx, mara.mtart, mara.meins
    FROM ckmlhd AS hd
    INNER JOIN mara ON mara.mandt = hd.mandt AND mara.matnr = hd.matnr
    LEFT OUTER JOIN makt ON makt.mandt = hd.mandt AND makt.matnr = hd.matnr
         AND makt.spras = session_context('LOCALE_SAP')
   WHERE hd.mandt = session_context('CLIENT')
     AND hd.bwkey = :p_bwkey;
  • 1~2행 — kalnr(ML 식별자)과 자재 키(matnr/bwkey/bwtar), 그리고 최종 출력에 쓸 마스터 속성(내역·유형·단위)을 한 번에 수집.
  • INNER JOIN mara — 자재 마스터 없는 kalnr은 의미 없으므로 inner. 조인에 mandt 포함(클라이언트 정합).
  • LEFT OUTER JOIN makt — 내역은 없을 수도 있으니 left. session_context('LOCALE_SAP')로 로그인 언어의 내역만.
  • WHERE — 클라이언트 필터(AMDP는 자동 필터 없음! 체크리스트 ⑥) + 파라미터 :p_bwkey를 가장 먼저 적용(필터 조기, 체크리스트 ③). 이후 모든 블록이 이 결과와 조인하므로 전체 데이터량이 여기서 결정됩니다.

2lt_mast1 — 최종 조인용 자재수준 집약 (2026-07-13 수정 ①)

lt_mast1 =
  SELECT mandt, matnr, bwkey, bwtar,
         MAX(maktx) AS maktx, MAX(mtart) AS mtart, MAX(meins) AS meins
    FROM :lt_master
   GROUP BY mandt, matnr, bwkey, bwtar;
✗ 수정 전 — 팬아웃 버그

최종 RETURN이 :lt_master(kalnr 수준)와 직접 조인 → TG14처럼 kalnr 10개인 자재는 동일 값 10행 출력(실측: 215행, TG14 ×10 — 합계 시 10배 과대).

✓ 수정 후

자재 수준으로 GROUP BY한 lt_mast1을 최종 조인에 사용 → 197행, 중복 0, 값 보존(TG14 opn 16 / cls 16 / 203.90).

  • 왜 MAX인가 — 같은 자재의 세그먼트들은 maktx/mtart/meins가 동일하므로 MAX는 사실상 "아무거나 하나"의 결정적 표현.
  • 왜 lt_master를 없애지 않나 — 잔액·이동 집계(블록 3·4)는 kalnr 수준 조인이 맞습니다(세그먼트 전부 합산해야 자재 총량). 문제는 오직 "이미 자재 수준으로 집계된 결과"에 kalnr 수준 마스터를 또 조인한 마지막 단계였습니다. 집계 수준과 조인 수준을 맞추는 것 — 이것이 이 버그의 교훈입니다.

3lt_bal — 기초(AB)/기말(EB) 잔액

lt_bal =
  SELECT ml.mandt, mas.matnr, mas.bwkey, mas.bwtar, ml.waers,
    SUM( CASE WHEN ml.categ = 'AB' AND ml.poper = :p_poper_from
              THEN ml.lbkum ELSE 0 END ) AS opn_qty,
    SUM( CASE WHEN ml.categ = 'AB' AND ml.poper = :p_poper_from THEN
           CASE WHEN :p_val_type = 'A'
                THEN ml.salk3 + ml.estprd + ml.estkdm + ml.mstprd + ml.mstkdm
                ELSE ml.salk3 END
         ELSE 0 END ) AS opn_amt,
    SUM( CASE WHEN ml.categ = 'EB' AND ml.poper = :p_poper_to
              THEN ml.lbkum ELSE 0 END ) AS cls_qty,
    SUM( CASE WHEN ml.categ = 'EB' AND ml.poper = :p_poper_to THEN
           CASE ... END ELSE 0 END ) AS cls_amt          -- (opn_amt와 동일 공식)
  FROM fcml_rep_v AS ml
  INNER JOIN :lt_master AS mas
    ON mas.mandt = ml.mandt AND mas.kalnr = ml.kalnr_mat
  WHERE ml.mandt = session_context('CLIENT')
    AND ml.bdatj = :p_gjahr  AND ml.curtp = :p_curtp
    AND ml.rldnr = :p_rldnr  AND ml.categ IN ('AB','EB')
    AND ml.poper IN (:p_poper_from, :p_poper_to)
  GROUP BY ml.mandt, mas.matnr, mas.bwkey, mas.bwtar, ml.waers;
  • 한 번의 스캔으로 4칼럼 — AB(from)와 EB(to)를 각각 SELECT하지 않고, 조건부 SUM(CASE) 4개로 한 번에 뽑습니다(조건부 집계 패턴).
  • 중첩 CASE = 평가 방식 — :p_val_type='A'면 표준가(salk3)에 ML 차이 4종(단층/다층 × 가격/환율)을 가산한 실제원가, 아니면 salk3만.
  • rldnr 필터 — 병렬 원장(0L 외)이 있으면 같은 이동이 원장 수만큼 중복 존재. 단일 원장 고정으로 이중집계 차단.
  • GROUP BY로 세그먼트 합산 — kalnr 수준으로 조인한 뒤 자재 수준으로 접습니다(블록 2 해설 참조).

4lt_doc — 기간 내 이동 라인 (2026-07-13 수정 ② 1/2)

lt_doc =
  SELECT doc.mandt, mas.matnr, mas.bwtar, mas.bwkey, doc.waers,
         doc.categ, doc.ptyp, ifnull(doc.glvor,'') AS glvor,
         ifnull(doc.runref,'') AS runref,
         doc.awtyp, doc.awref, doc.aworg, doc.awitem,
         doc.quant AS move_qty,
         CASE WHEN :p_val_type = 'A'
              THEN doc.stval + doc.prd + doc.kdm
              ELSE doc.stval END AS move_amt
  FROM mldoc AS doc
  INNER JOIN :lt_master AS mas
    ON mas.mandt = doc.mandt AND mas.kalnr = doc.kalnr
  WHERE doc.mandt = session_context('CLIENT')
    AND doc.curtp = :p_curtp  AND doc.rldnr = :p_rldnr
    AND doc.jahrper BETWEEN :p_gjahr || :p_poper_from
                        AND :p_gjahr || :p_poper_to
    AND doc.categ IN ('ZU','VN','PC');
  • ifnull — glvor/runref는 NULL 가능. 매핑 비교('=' 또는 '*')가 NULL에 빠지지 않도록 ''로 정규화.
  • awtyp/awref/aworg/awitem을 실어 나름 — 다음 블록(matdoc 제한 조인)의 열쇠. 이 블록을 분리한 것이 수정 ②의 핵심: 필터된 이동 라인을 먼저 확정해야 matdoc을 그 참조분만 읽을 수 있습니다.
  • jahrper 범위 — :p_gjahr || :p_poper_from 문자열 연결로 '2020'+'001'='2020001'. NUMC 7자리끼리의 BETWEEN이라 안전.
  • categ IN (ZU,VN,PC) — 입고/출고/가격변경만. AB/EB(잔액)는 블록 3의 몫.
  • move_amt 공식 — 잔액과 동일 사상: 'A'면 stval+prd+kdm(가격·환율차 포함), 아니면 stval. 잔액과 이동의 평가 기준이 같아야 diff=0이 성립합니다.

5lt_mat — MATDOC 중복 제거 + 참조분 제한 (수정 ② 2/2)

lt_mat =
  SELECT mat.mandt, mat.mblnr, mat.mjahr, mat.zeile,
         MAX(mat.bwart) AS bwart, MAX(mat.shkzg) AS shkzg, MAX(mat.sobkz) AS sobkz
    FROM matdoc AS mat
    INNER JOIN ( SELECT DISTINCT mandt, awref, aworg, awitem
                   FROM :lt_doc
                  WHERE awtyp = 'MKPF' ) AS d
      ON  d.mandt = mat.mandt
      AND d.awref = mat.mblnr
      AND CAST( mat.mjahr AS INTEGER ) = CAST( d.aworg  AS INTEGER )
      AND CAST( mat.zeile AS INTEGER ) = CAST( d.awitem AS INTEGER )
   GROUP BY mat.mandt, mat.mblnr, mat.mjahr, mat.zeile;
  • 왜 dedup? — 함정 ②(2부): MATDOC 물리 키는 GUID라 (mblnr,mjahr,zeile)가 유일하지 않음. GROUP BY로 라인당 1행 보장. 같은 라인의 분할 행들은 bwart/shkzg가 동일하므로 MAX()가 결정적(아무거나 하나 = 그 값).
  • 왜 INNER JOIN 서브쿼리? — 수정 전에는 클라이언트 전체 matdoc(43,711행, 운영이면 수천만)을 dedup했습니다. 지금은 lt_doc이 실제 참조하는 문서 키(DISTINCT awref/aworg/awitem)로 제한 — 체크리스트 ③(필터 조기)의 전형적 적용.
  • CAST 양쪽 통일 — aworg(CHAR)와 mjahr(NUMC), awitem과 zeile은 타입이 달라 INTEGER로 맞춰 비교(체크리스트 ⑧ 암묵 형변환 방지). 서브쿼리가 awtyp='MKPF'만 남기므로 aworg는 항상 연도 숫자 — CAST 안전.

6lt_move — 이동 라인 + 자재문서 속성 결합

lt_move =
  SELECT
    ROW_NUMBER() OVER () AS rid,
    src.mandt, src.matnr, src.bwtar, src.bwkey, src.waers,
    src.categ, src.ptyp, src.glvor, src.runref,
    ifnull(mat.bwart,'') AS bwart,
    ifnull(mat.shkzg,'') AS shkzg,
    ifnull(mat.sobkz,'') AS sobkz,
    src.move_qty, src.move_amt
  FROM :lt_doc AS src
  LEFT OUTER JOIN :lt_mat AS mat
    ON  src.awtyp = 'MKPF'
    AND mat.mandt = src.mandt
    AND mat.mblnr = src.awref
    AND CAST( mat.mjahr AS INTEGER ) =
        CASE WHEN src.awtyp = 'MKPF' THEN CAST( src.aworg AS INTEGER ) ELSE -1 END
    AND CAST( mat.zeile AS INTEGER ) =
        CASE WHEN src.awtyp = 'MKPF' THEN CAST( src.awitem AS INTEGER ) ELSE -1 END;
  • ROW_NUMBER() OVER () AS rid — 라인마다 고유 번호. ORDER 없는 ROW_NUMBER는 순서 비결정이지만 여기선 식별자로만 쓰므로 무해(블록 7의 PARTITION BY 기준).
  • LEFT OUTER JOIN — 자재문서가 없는 라인(송장 RMRP, 가격변경 PC 등)도 살려야 하므로 left. 그 경우 bwart/shkzg/sobkz는 ifnull로 ''.
  • CASE 가드 — src.awtyp='MKPF'가 이미 조인 조건에 있지만, HANA가 조건 평가 순서를 보장하지 않으므로 CAST(src.aworg)가 비MKPF 행(비숫자 aworg)에서 터지지 않도록 CASE로 감쌉니다. 방어적 형변환의 교과서 사례.

7lt_ranked — 매핑 규칙 적용 (분류의 심장)

lt_ranked =
  SELECT rid, mandt, matnr, bwtar, bwkey, waers, move_qty, move_amt, cat_code
  FROM (
    SELECT src.rid, ..., src.move_qty, src.move_amt,
           ifnull( map.cat_code,
                   CASE WHEN src.categ = 'ZU' THEN 'IN_OTH'
                        WHEN src.categ = 'VN' THEN 'OUT_OTH'
                        WHEN src.categ = 'PC' AND src.runref <> '' THEN 'ADJ_ML'
                        WHEN src.categ = 'PC' THEN 'ADJ_REV'
                        ELSE 'ADJ_REV' END ) AS cat_code,
           ROW_NUMBER() OVER ( PARTITION BY src.rid ORDER BY map.seqnr ASC ) AS rk
    FROM :lt_move AS src
    LEFT OUTER JOIN ytco_slmap AS map
      ON  map.mandt = src.mandt
      AND ( map.categ = src.categ OR map.categ = '*' )
      AND ( map.ptyp  = src.ptyp  OR map.ptyp  = '*' )
      AND ( map.glvor = src.glvor OR map.glvor = '*' )
      AND ( map.bwart = src.bwart OR map.bwart = '*' )
      AND ( map.shkzg = src.shkzg OR map.shkzg = '*' )
      AND ( map.sobkz = src.sobkz OR map.sobkz = '*' )
  )
  WHERE rk = 1;
  • 와일드카드 조인 — 6개 차원 각각 "정확히 일치 또는 매핑값이 '*'". 한 라인이 여러 매핑 행에 동시 매칭될 수 있습니다(예: 101/S 라인은 1000·1001·1010 모두 후보).
  • seqnr 최소 우선 — ROW_NUMBER() OVER (PARTITION BY rid ORDER BY seqnr)로 라인(rid)마다 매칭 행들에 순위를 매기고 rk=1만 남김 → "구체적 규칙을 작은 seqnr에 두면 이긴다"는 운영 규칙이 성립(6부).
  • 폴백 CASE — 어떤 매핑에도 안 걸리면(map.cat_code IS NULL) categ 기준 안전망: ZU→IN_OTH, VN→OUT_OTH, PC는 runref 유무로 ADJ_ML(결산 정산)/ADJ_REV(수동 가격변경) 분기. 어떤 라인도 버려지지 않으므로 diff=0(질량 보존)이 구조적으로 보장됩니다.
  • rid가 필요한 진짜 이유 — PARTITION 기준이 문서 키가 아니라 rid인 것은, 같은 문서라도 라인별로 독립 분류되어야 하기 때문입니다.

8lt_pivot — 카테고리 → 칼럼 피벗

lt_pivot =
  SELECT mandt, matnr, bwtar, bwkey, MAX(waers) AS waers,
    SUM( CASE WHEN cat_code='IN_PO'  THEN move_qty ELSE 0 END ) AS in_po_qty,
    SUM( CASE WHEN cat_code='IN_PO'  THEN move_amt ELSE 0 END ) AS in_po_amt,
    ... (in_* 8쌍) ...
    SUM( CASE WHEN cat_code='OUT_SLS' THEN -move_qty ELSE 0 END ) AS out_sls_qty,
    ... (out_* 10쌍, 전부 -move_*) ...
    SUM( CASE WHEN cat_code='ADJ_PPV' THEN move_amt ELSE 0 END ) AS adj_ppv_amt,
    SUM( CASE WHEN cat_code='ADJ_REV' THEN move_amt ELSE 0 END ) AS adj_rev_amt,
    SUM( CASE WHEN cat_code='ADJ_ML'  THEN move_amt ELSE 0 END ) AS adj_ml_amt
  FROM :lt_ranked
  GROUP BY mandt, matnr, bwtar, bwkey;
  • 조건부 SUM 피벗 — cat_code(세로)를 40여 개 칼럼(가로)으로. 표준 SQL만으로 피벗하는 정석 패턴.
  • 부호 규약 구현 지점 — out_*만 -move_qty/-move_amt: VN 라인은 원래 음수이므로 −를 곱해 양수 표기. 취소로 생긴 반대 부호도 같은 칼럼에서 자동 상계.
  • adj_*는 금액만 — 가격변경(PC)은 수량 이동이 없으므로 qty 칼럼 자체가 없고, 최종 출력에서 0으로 캐스트(블록 10).

9lt_keys — 결과 드라이버

lt_keys =
  SELECT mandt, matnr, bwtar, bwkey FROM :lt_bal
  UNION
  SELECT mandt, matnr, bwtar, bwkey FROM :lt_pivot;
  • 왜 UNION(중복 제거)? — "잔액만 있는 자재"(이동 없음)도 "이동만 있는 자재"(기중 입·출고 후 잔액 0)도 한 행씩 나와야 합니다. 어느 한쪽만 드라이버로 삼으면 반대 케이스가 누락됩니다.
  • 여기서는 중복 제거가 목적이므로 UNION ALL이 아닌 UNION이 맞습니다(체크리스트 ⑪의 예외 케이스).

10RETURN — 최종 조립

RETURN
  SELECT
    k.mandt AS client, :p_gjahr AS gjahr, :p_poper_from AS poper_from, ...,
    mas.maktx, mas.mtart, mas.meins,
    coalesce(bal.waers, pv.waers) AS waers,
    ifnull(bal.opn_qty,0) AS opn_qty, ...
    CAST(0 AS "$ABAP.type( ml4h_lbkum )") AS adj_ppv_qty, ...
    ( ifnull(bal.opn_qty,0)
      + ifnull(pv.in_po_qty,0) + ... (in 8종)
      - ifnull(pv.out_sls_qty,0) - ... (out 10종)
      - ifnull(bal.cls_qty,0) ) AS diff_qty,
    ( ... 금액은 + adj 3종까지 가산 ... ) AS diff_amt
  FROM :lt_keys AS k
  INNER JOIN :lt_mast1 AS mas
    ON mas.mandt = k.mandt AND mas.matnr = k.matnr AND mas.bwtar = k.bwtar AND mas.bwkey = k.bwkey
  LEFT OUTER JOIN :lt_bal  AS bal ON (4키 동일)
  LEFT OUTER JOIN :lt_pivot AS pv ON (4키 동일);
  • INNER JOIN :lt_mast1 — 수정 ①의 결과. 자재 수준 드라이버(lt_keys)에 자재 수준 마스터를 조인 — 수준이 일치하므로 팬아웃 없음.
  • LEFT JOIN bal/pv + ifnull(…,0) — 잔액만/이동만 있는 자재의 반대편 값을 0으로. NULL이 산식에 들어가면 결과 전체가 NULL이 되므로 필수.
  • coalesce(bal.waers, pv.waers) — 통화는 잔액 우선, 없으면 이동에서.
  • CAST(0 AS "$ABAP.type( ml4h_lbkum )") — adj_*_qty는 개념상 0인데, TF 반환 구조와 타입을 정확히 맞추기 위해 ABAP 사전 타입을 SQLScript에서 참조하는 특수 문법을 사용.
  • diff 산식 — 3부 부호 규약 그대로: 기초 + Σ입고 − Σ출고(양수 표기이므로 빼기) − 기말. 금액은 조정 3종을 더한 뒤 기말을 뺍니다. 이 식이 0이 되는 것이 수불부의 자기 검증입니다.

6.4 새 이동유형 추가 워크스루

예: 이동유형 561(기초재고 인수, S)이 새로 쓰이기 시작해 IN_OTH로 잡히는 경우 —
