1부. 왜 CDS인가

1.1 Code Pushdown 기본

전통적인 ABAP 프로그램은 데이터베이스에서 데이터를 가져와서(SELECT) 애플리케이션 서버의 ABAP 코드(LOOP, 계산, 집계)에서 가공했습니다. 데이터가 많을수록 DB↔AS 사이를 오가는 비용이 커집니다.

[전통 방식]  DB  ──(많은 행 전송)──▶  ABAP AS  ──LOOP/계산/집계──▶ 결과
[CDS 방식]   DB  ──조인·계산·집계를 DB에서 수행──▶ 결과(적은 행)만 AS로

Code Pushdown은 "계산을 데이터가 있는 곳(DB)으로 내린다"는 원칙입니다. SAP HANA는 인메모리·컬럼 기반 DB라 조인/집계가 매우 빠르므로, 로직을 DB 가까이(=CDS View)에 두는 것이 유리합니다. CDS는 이 원칙을 구현하는 SAP의 표준 데이터 모델링 수단입니다.

1.2 SE11 클래식 뷰 vs CDS 뷰 기본

1.3 CDS는 어디에 사는가 기본

CDS는 DDL 소스(Data Definition Language)로 작성합니다. 편집은 SE80이 아니라 ADT(ABAP Development Tools for Eclipse)에서 합니다. 활성화하면 내부적으로 DB에 실제 뷰(또는 HANA 뷰)가 생성되고, ABAP Open SQL에서 SELECT ... FROM 뷰이름으로 바로 사용할 수 있습니다.

용어 — 최신 문법은 DEFINE VIEW ENTITY(뷰 엔티티)입니다. 예전 DEFINE VIEW ... AS SELECT(클래식 CDS 뷰, DDIC 뷰를 함께 생성)와 구분하세요. 본 가이드는 View Entity를 기본으로 합니다. 예제 01 ▸

1.4 ECC에서는 테이블, S/4HANA에서는 CDS 뷰 기본

아직 ECC를 쓰는 현장이 많습니다. ECC에서 익숙하던 여러 테이블이 S/4HANA에서는 테이블이 아니라 CDS 뷰(호환성/리다이렉트 뷰)로 바뀌었습니다. 왜 이렇게 되었는지부터 이해하면 데이터를 찾을 때 헷갈리지 않습니다.

배경 ① HANA 컬럼 스토어의 구조적 이점

HANA는 데이터를 컬럼 단위(Column Store)로 저장·압축합니다. 로우 스토어와 달리, 쿼리는 실제로 접근하는 컬럼만 읽습니다. 따라서 테이블이 수백 개 필드로 넓어져도(wide table) 조회 비용이 비례해서 늘지 않습니다. 이 덕분에 여러 테이블을 하나의 넓은 통합 테이블(예: ACDOCA는 약 300개 이상 필드)로 합치는 설계가 가능해졌습니다.

배경 ② 집계·인덱스 테이블 불필요

컬럼 스토어의 고속 스캔·압축 덕분에 별도의 총계 테이블(GLT0, FAGLFLEXT)이나 보조 인덱스 테이블(BSIS 등)을 유지할 필요가 없어졌습니다. 합계는 조회 시점에 실시간(on-the-fly)으로 집계합니다.

배경 ③ Insert-only + 통합 문서 테이블

데이터 중복을 줄이고, 하나의 통합 문서 테이블로 정합성과 조회 성능을 동시에 확보합니다. 결론은 "저장은 통합 wide 테이블로, 소비는 목적별 CDS 뷰로 얇게 본다" — 이것이 S/4HANA 데이터 모델 철학입니다. 기존 테이블명은 호환성(compatibility)/리다이렉트 CDS 뷰로 유지되어 예전 코드도 계속 동작합니다.

2부. CDS 기본 문법

2.1 View Entity 기본 골격 기본

// 헤더 어노테이션
@AccessControl.authorizationCheck: #NOT_REQUIRED
@EndUserText.label: 'EDU 01 - Basic Flight View'
define view entity ZI_JNC_EDU_01_FLIGHT
  as select from sflight
{
  key carrid    as AirlineID,
  key connid    as ConnectionID,
  key fldate    as FlightDate,
      price     as Price,
      currency  as CurrencyCode,
      seatsmax  as SeatsMax,
      seatsocc  as SeatsOccupied
}

define view entity 뒤에 이름, as select from 뒤에 소스(테이블/다른 뷰), 중괄호 안에 필드 목록을 씁니다. key는 키 필드, as는 별칭입니다.

클래식과의 차이 — 클래식 CDS 뷰는 @AbapCatalog.sqlViewName(별도 DDIC 뷰 이름)이 필요했지만, View Entity는 필요 없습니다. 이름 하나로 관리됩니다. 예제 01 ▸

2.2 SELECT 리스트 · 별칭 · key · 세션 변수 기본

필드는 소스필드 as 별칭 형태로 노출합니다. 별칭은 관례상 CamelCase(AirlineID)를 씁니다. 세션 변수로 현재 문맥 값을 가져올 수 있습니다.

세션 변수의미
$session.system_date현재 시스템 날짜
$session.user현재 사용자
$session.client현재 클라이언트

2.3 계산 필드 · CASE · CAST 기본

  // 좌석 점유율(%) — division-by-zero를 CASE로 방어하고 dec(15,2)로 반올림
  case seatsmax
    when 0 then cast( 0 as abap.dec(15,2) )
    else cast( cast( seatsocc as abap.dec(15,2) )
               / seatsmax * 100 as abap.dec(15,2) )
  end as OccupancyRate,
  case
    when seatsocc >= seatsmax then 'FULL'
    when seatsocc = 0         then 'EMPTY'
    else                           'AVAILABLE'
  end as SeatStatus

CASE는 단순형(case field when v then ...)과 검색형(case when 조건 then ...) 두 가지가 있습니다. CAST는 타입 변환입니다. 정수끼리 나누면 정수가 되므로, 소수 결과가 필요하면 먼저 cast(... as abap.dec) 해야 합니다. 예제 02 ▸

2.4 JOIN & UNION 기본

  as select from sflight as f
  inner join      scarr as c on c.carrid = f.carrid
  left outer join spfli as p on p.carrid = f.carrid
                              and p.connid = f.connid

inner join은 양쪽 다 매칭되는 행만, left outer join은 왼쪽은 모두 유지하고 오른쪽 없으면 NULL을 채웁니다. UNION ALL은 두 SELECT 결과를 이어붙입니다(컬럼 개수·타입이 일치해야 함).

주의 — View Entity에서 UNION을 쓰면 @Metadata.ignorePropagatedAnnotations: true 어노테이션이 필요합니다. 예제 03(JOIN) ▸ 예제 05(UNION) ▸

2.5 집계 · GROUP BY · HAVING 기본

{
  key carrid as AirlineID,
      count( * )      as FlightCount,
      sum( seatsocc ) as TotalOccupied,
      avg( cast( price as abap.dec(15,2) ) as abap.dec(15,2) ) as AvgPrice
}
group by carrid
having count( * ) > 1

SUM/AVG/COUNT/MAX/MIN으로 집계합니다. 집계하지 않는 필드는 반드시 GROUP BY에 있어야 합니다. HAVING은 집계 결과에 대한 필터입니다.

실전 함정 — 통화(CURR) 필드의 AVG는 대상 타입으로 DEC를 직접 지정할 수 없습니다. 먼저 cast(price as abap.dec(..))로 통화 성격을 벗긴 뒤 평균내야 합니다. 예제 04 ▸

2.6 파라미터 뷰 기본

define view entity ZI_JNC_EDU_06_PARAM
  with parameters
    P_MinSeats : abap.int4
  as select from sflight
{ ... }
where seatsmax >= $parameters.P_MinSeats

파라미터는 with parameters로 선언하고, 본문에서 $parameters.이름으로 참조합니다(옛 :파라미터 표기는 View Entity에서 오류). 호출은 SELECT ... FROM view( P_MinSeats = 380 ). 예제 06 ▸

2.7 표현식 심화 · 통화/수량 시맨틱 심화

금액·수량 필드는 어떤 통화/단위 필드와 짝인지 어노테이션으로 알려주면 프레임워크가 올바르게 처리·표시합니다.

3부. Association

3.1 Association이란 — JOIN과의 차이 기본

Association은 뷰 사이의 관계를 미리 정의해 두고, 실제로 관련 필드를 사용할 때만 조인이 일어나게 하는 "지연 조인(lazy join)"입니다. 관계를 재사용·노출할 수 있어 데이터 모델을 조합하기 좋습니다.

[JOIN]         선언 즉시 조인 실행 → 안 쓰는 컬럼도 조인 비용 발생
[ASSOCIATION]  관계만 정의 → 경로식(_Carrier.carrname)을 쓸 때만 조인

3.2 정의와 사용 기본

  as select from sflight
  association [1..1] to scarr as _Carrier
    on $projection.AirlineID = _Carrier.carrid
{
  key carrid as AirlineID,
      _Carrier.carrname as CarrierName,  // 경로식 사용
      _Carrier                             // association 자체 노출(재사용용)
}

이름은 관례상 밑줄로 시작(_Carrier). on 조건에서 자기 뷰 필드는 $projection.별칭으로 참조합니다. 필드 목록에서 _Carrier.carrname처럼 경로식을 쓰거나, _Carrier만 노출해 상위 뷰에서 재사용하게 할 수 있습니다. 예제 07 ▸ 예제 08(재사용) ▸

3.3 카디널리티 기본

표기의미
[1..1]정확히 1건 (예: 항공편 → 항공사)
[0..1]0 또는 1건
[0..*] / [1..*]여러 건 (to-many, 예: 항공편 → 예약들)
to parent자식→부모(컴포지션 관계)

3.4 필터드 Association · 중첩 경로식 심화

  // 특정 예약만 골라 스칼라 값을 가져오기

4부. VDM 계층 & 네이밍 표준

4.1 Basic / Composite / Consumption 계층 기본

SAP는 CDS를 목적에 따라 계층으로 나눕니다. 이를 VDM(Virtual Data Model)이라 합니다.

 Consumption (소비)   ← 앱/OData 지향, 필드 선별·라벨 정리   @VDM.viewType: #CONSUMPTION
        ▲
 Composite (복합)     ← 여러 기반 뷰를 결합, 파생 필드 계산   @VDM.viewType: #COMPOSITE
        ▲
 Basic (기반)         ← 테이블 1:1 투영 + association만       @VDM.viewType: #BASIC
        ▲
   [ 데이터베이스 테이블 ]

기반(Basic) 뷰는 로직을 최소화하고, 위 계층으로 갈수록 조합·가공이 늘어납니다. 아래층을 재사용하므로 중복이 줄고 유지보수가 쉬워집니다. 예제 11~13(3계층) ▸

4.2 @VDM.viewType · 인터페이스 뷰 기본

SAP 표준 재사용 뷰(공개 계약)는 인터페이스 뷰라 하며 I_로 시작합니다(예: I_Currency). 이들은 released(안정) 상태로 고객이 참조해도 안전합니다. 내부 전용 뷰는 P_(private)로 두어 외부 참조를 막습니다.

4.3 재사용 설계 원칙 심화

같은 계산·조인을 여러 곳에서 반복하지 말고 기반/복합 뷰로 한 번 정의해 재사용하세요. 계층을 나누면 ① 변경 영향 범위가 작아지고 ② 각 뷰가 한 가지 책임만 지며 ③ 테스트·이해가 쉬워집니다.

CDS 네이밍 표준 기본

SAP는 한 글자 역할 접두사를 예약합니다. 고객은 앞에 Z/Y를 붙여 씁니다.

접두사역할계층
I_Interface view (재사용·공개, released)재사용
C_Consumption view (앱/OData 소비)소비
P_Private view (내부 전용)기반
R_Root/Restricted (RAP 루트, 문맥 의존)기반
E_ / X_Extend / Extension include확장
TP_Transactional Projection (RAP 소비)소비

→ 고객 뷰 예: ZI_FLIGHT, ZC_FLIGHT, ZP_FLIGHT.

RAP 마법사 자동 접두사

ADT의 "Generate ABAP Repository Objects"(테이블에서 RAP BO 생성) 마법사가 제안하는 기본 접두사:

오브젝트기본 접두사
Data Definition (base/interface, root view)ZR_
Projection View (consumption)ZC_
Metadata ExtensionZC_...
Service DefinitionZUI_ / Z..._O2

즉 RAP 관례는 사실상 ZR_(기반) → ZC_(투영/소비) 쌍입니다.

요소·기타 관례

  • 필드: CamelCase (AirlineID, SeatsOccupied)
  • 파라미터: P_ 접두사 (P_DisplayCurrency)
  • Association: _로 시작 (_Carrier, _Flights)
  • DCL(Access Control): 보호 대상 뷰명과 동일 이름 권장
주의 — Connection, Client 등은 예약어라 요소 이름으로 쓸 수 없습니다. ConnectionID처럼 우회하세요. (본 교육 예제는 ZI_JNC_EDU_##_ 네이밍을 씁니다 — JNC는 이번 예제 한정 태그입니다.)

5부. 어노테이션 & 심화 기능 (총망라)

5.1 어노테이션 개념 · 스코프 · 전파 기본

어노테이션은 @로 시작하는 메타데이터입니다. 엔티티 스코프(뷰 전체, 헤더에 위치)와 요소 스코프(개별 필드 위에 위치)가 있습니다. 뷰를 재사용하면 요소 어노테이션이 상위로 전파됩니다. UNION 등에서는 @Metadata.ignorePropagatedAnnotations로 전파를 끊습니다.

5.2 @Analytics (CUBE/DIMENSION) 심화

@Analytics.dataCategory: #CUBE
define view entity ZI_JNC_EDU_14_ANALYTICS as select from sflight { ...
  @DefaultAggregation: #SUM
  seatsocc as SeatsOccupied,
  @Semantics.amount.currencyCode: 'CurrencyCode'
  @DefaultAggregation: #SUM
  paymentsum as PaymentSum, ... }

#CUBE는 측정값(measure)을 담는 분석 뷰, #DIMENSION은 마스터/차원 뷰입니다. @DefaultAggregation으로 측정값의 기본 집계를 지정합니다. 예제 14 ▸

5.4 CDS Extend 심화

extend view entity ZI_JNC_EDU_01_FLIGHT with
{
  sflight.paymentsum as PaymentSum
}

원본 뷰를 수정하지 않고 필드를 추가합니다. 대상 뷰는 @AbapCatalog.viewEnhancementCategory: [#PROJECTION_LIST]를 가져야 하며, 추가 필드에는 소스 테이블 프리픽스(sflight.)가 필요합니다. 예제 15 ▸

5.5 Access Control / DCL 심화

@AccessControl.authorizationCheck: #CHECK  // 뷰 쪽
...
// DCL 오브젝트(뷰와 동일 이름)
@MappingRole: true
define role ZI_JNC_EDU_16_DCL {
  grant select on ZI_JNC_EDU_16_DCL
    where ( AirlineID ) = 'AA';
}

뷰에 #CHECK를 걸고, 같은 이름의 DCL(Data Control Language) 역할에서 행 수준 접근을 제한합니다. 실무에서는 리터럴 대신 aspect pfcg_auth(...)로 PFCG 권한객체와 연동합니다. 예제 16 ▸

5.6 Table Function + AMDP 심화

define table function ZI_JNC_EDU_17_TABFUNC
  returns { client : abap.clnt; airlineid : s_carr_id; ...; }
  implemented by method ZCL_JNC_EDU_17_TABFUNC=>get_flight_stats;

순수 CDS로 표현하기 어려운 로직(복잡한 SQLScript)은 Table Function으로 선언하고, 실제 구현을 AMDP(ABAP Managed Database Procedure, HANA SQLScript) 메서드에 둡니다. client-specific이면 RETURNS 첫 필드에 client : abap.clnt가 필요합니다. 예제 17 ▸

5.7 계층 CDS (DEFINE HIERARCHY) 심화

define hierarchy ZI_JNC_EDU_18_HIER
  as parent child hierarchy(
    source ZI_JNC_EDU_18_HIERSRC
    child to parent association _Parent
    start where ParentId = ' '
    siblings order by NodeId
  )
{ key NodeId, ParentId, NodeName, NodeType }

부모-자식 자기참조 association을 가진 소스로 계층을 정의합니다. 활성화하면 계층 순위/레벨 같은 pseudo-column을 계층 소비 시 사용할 수 있습니다. 예제 18 ▸
