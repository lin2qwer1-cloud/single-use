1부. 원천추적이란 무엇인가 업무

1.1 문제 상황

코스트센터 회계(CO-CCA)에서는 월말에 배부 사이클이 돌아갑니다. 청소팀 비용이 각 부서로 나뉘고(배분, distribution), 간접부문 비용이 어세스먼트 원가요소로 뭉쳐 생산부문으로 넘어가고(어세스먼트, assessment), 그 생산부문 비용은 다시 CO-PA나 오더 정산으로 CO 밖으로 나갑니다. 사이클이 서너 단계만 겹쳐도 최종 리시버 입장에서는 "내 계정에 앉아 있는 이 금액이 애초에 어디서 온 돈인지"를 표준 리포트로는 알 수 없게 됩니다.

 청소팀(1차비용 1,000) ──배분──> 전산실 ──어세스먼트──> 생산라인 ──정산/CO-PA──> CO 밖
                                    ↑
                     "생산라인에 앉은 금액 중 청소팀 몫은 얼마인가?"  ← 이 질문에 답하는 엔진

1.2 엔진이 답하는 것

한 회계월의 배부 전기를 실행된 순서 그대로 재생하여, 모든 최종 잔액·유출액을 (원천 오브젝트, 원천 원가요소) → (최종 오브젝트, 최종 원가요소) 조합의 전수 테이블로 분해합니다. 결과 한 행은 이렇게 읽습니다: "KSZ004X003의 825900 계정에서 발생한 3,823.53이, 지금 KSZ004Z002의 1000100 계정에 앉아 있다."

1.3 두 가지 설계 결정 개발

  • 사이클 마스터(T811C/T811S)는 참조하지 않는다 — 마스터는 사후 변경될 수 있습니다. 이 엔진은 실제 전기 이력인 T811D(사이클 실행 인덱스)와 CO 명세서(COEP/COEJ)만 믿습니다. "그때 실제로 일어난 일"만이 근거입니다.
  • 버리는 금액이 없다 — 프루닝으로 계산에서 제외한 잔량도 PRUNED 행으로, 원천을 모르는 금액도 UNTRACEABLE 행으로 이름을 붙여 출력합니다. 그래서 들어온 총량(in)과 나간 총량(out)이 항상 일치합니다 — 대사가 검사가 아니라 설계 불변식이 됩니다.

2부. 원천 테이블 & ERD

2.1 ERD — 조인 관계 한눈에 개발

  • 파란 상자(스파인)가 계산의 원천, 회색 상자는 조인/표시용입니다. 초록 상자가 이 문서의 주인공입니다.
  • T811D↔COEP는 1:N(한 사이클 run 전표에 여러 명세 라인), COBK↔COEP/COEJ도 1:N(헤더:라인).
  • CSKS/CSKT/AUFK/PRPS는 엔진이 아니라 헬퍼 클래스가 OBJNR을 풀어 표시 속성을 얻을 때만 씁니다. 관계선이 점선인 곳은 비식별(선택적) 관계입니다.

2.2 테이블별 사용 명세

테이블역할엔진이 쓰는 필드핵심 필터
T811D"실제로 전기된 사이클 run"의 앵커. 이 인덱스에 없는 전표는 사이클 배부가 아님 nkrsub(KOKRS), gjahr, perio, docnr(전표번호), cycle, reversed tab='CCSS'(코스트센터 사이클), reversed=''
COEP실적 명세 라인 — 배부 엣지(리시버 라인)와 자기비용의 원천 vrgng, parob(파트너=sender), objnr(=receiver), kstar, wkgbtr, beknz wrttp='04', versn='000'
COEJ계획 명세 — 연간 1행에 wkg001~012 기간 피벗. 언피벗해서 사용 COEP와 동일 + wkg001..wkg012 wrttp='01', versn=파라미터
COBK전표 헤더 — 실행시각과 역분개 판별 cpudt, cputm(→ 실행순서), stflg, stokz stflg='' AND stokz='' (실적 엣지)
CSKS/CSKT코스트센터 마스터/텍스트 (표시)bukrs, gsber, ktext키일자 = 조회월 말일
AUFK/PRPS내부오더/WBS 마스터 (표시)bukrs/pbukr, gsber/pgsbr, ktext/post1objnr 직접 조회
역분개는 이중으로 걸러야 한다 — run 레벨에서 T811D REVERSED='X', 전표 레벨에서 COBK STFLG(역분개 전표)/STOKZ(역분개된 전표). 어느 한쪽만 쓰면 재전기가 뒤엉킨 데이터(IDES 2018년 데이터가 그랬습니다)에서 유령 엣지가 살아남아 금액이 이중 계산됩니다.
S/4에서 COEP는 리다이렉트 테이블 — ABAP Open SQL로 SELECT ... FROM coep를 쓰면 커널이 자동으로 호환 뷰로 우회시켜 줍니다(그래서 이 엔진은 그냥 coep라고 씁니다). 반면 AMDP/네이티브 SQL·DB 콘솔에서는 리다이렉트가 없으므로 v_coep_view를 직접 읽어야 합니다. 같은 이름을 읽어도 경로에 따라 다른 데이터가 나올 수 있는 대표 함정입니다.

1make_exec_seq — 시각을 정수 하나로

METHOD make_exec_seq.
  DATA lv_dc TYPE c LENGTH 8.
  DATA lv_tc TYPE c LENGTH 6.
  lv_dc = iv_cpudt.                 " D(8) → C(8): '20231025'
  lv_tc = iv_cputm.                 " T(6) → C(6): '143210'
  rv_seq = CONV int8( lv_dc ) * 1000000 + CONV int8( lv_tc ).
ENDMETHOD.                          " → 20231025143210
  • 왜 문자 경유? — 날짜(D)·시간(T)을 바로 숫자 연산에 쓰면 내부 표현이 아니라 "달력 연산"이 됩니다. 문자로 받으면 YYYYMMDD/HHMMSS 자릿수 그대로 확보되고, 이를 이어붙인 정수는 시각순 = 크기순이 보장됩니다.
  • int8 — 20,231,025,143,210은 int4 최대값(약 21억)을 훌쩍 넘습니다. 8바이트 정수 필수.
  • 같은 초에 전기된 라인들은 같은 exec_seq → 4부 블록 6에서 "같은 묶음"으로 함께 처리됩니다.

2derive_rule — 엣지의 성격 판정

METHOD derive_rule.
  DATA(lv_parob) = condense( CONV string( iv_parob ) ).
  rv_rule = COND #( WHEN strlen( lv_parob ) <= 2 THEN 'U'
                    WHEN iv_vrgng = 'U'           THEN 'B'
                    ELSE 'A' ).
ENDMETHOD.
  • 1행 — PAROB(파트너 오브젝트)를 문자열로 바꿔 공백 제거. OBJNR은 'KS'+KOKRS+KOSTL 구조라 정상값은 최소 6자 이상입니다.
  • 2행: 길이≤2 → 'U' — 'KS'만 있거나 비어 있으면 sender를 특정할 수 없는 라인 → 추적불가로 정직하게 분류.
  • 3행: iv_vrgng='U' → 'B' — 호출부가 vrgng+3(1)(4번째 글자)를 넘깁니다. RKIU/RKPU(어세스먼트)의 4번째 글자가 'U' → 가중평균 규칙 B. RKIV/RKPV(배분)·RKIB(간접활동)는 ELSE → 계정보존 규칙 A. 네 글자를 다 비교하지 않고 판별 문자 하나만 받는 것이 이 설계의 트릭입니다.

6.1 in = out은 왜 항상 성립하는가 개발

 in  = Σ 자기비용 + Σ rule-U 유입 + Σ pool=0 폴백 유입      ← 유입은 전부 in에 기록
 out = Σ et_trace.amount
     = Σ (라운드마다 state에 남은 것 + PRUNED로 방출된 것)

 각 라운드에서:  이동분 + 잔류분 = 원래 금액   (비례 배분의 항등식)
                프루닝된 조각도 lt_out으로 이동할 뿐 소멸하지 않음
 따라서 유입된 모든 금액은 반드시 et_trace 어딘가에 존재  →  in = out

그래서 리포트 헤더의 diff가 0이 아니면 그것은 데이터 문제가 아니라 엔진 버그입니다. 반올림조차 버킷별 잔차보정으로 회수하므로 diff는 정확히 0.00이어야 합니다.
