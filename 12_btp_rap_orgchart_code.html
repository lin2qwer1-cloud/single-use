1부. 테이블 — 자기참조와 드래프트 기초

1.1 ZJMG_ORG_NODE — 실데이터 테이블

@EndUserText.label : 'Org Chart Node (JMG demo)'
@AbapCatalog.enhancement.category : #NOT_EXTENSIBLE
@AbapCatalog.tableCategory : #TRANSPARENT
@AbapCatalog.deliveryClass : #A
define table zjmg_org_node {
  key client      : abap.clnt not null;
  key node_id     : zjmg_org_node_id not null;
  parent_id       : zjmg_org_node_id;
  org_name        : zjmg_org_name;
  job_title       : zjmg_org_title;
  last_changed_at : timestampl;
}
줄주해
@AbapCatalog.enhancement.category확장 카테고리 선언. 이 시스템에서는 빼면 체크는 통과하는데 저장이 거부됩니다 — 테이블 DDL의 사실상 필수 어노테이션.
key node_id / parent_id둘 다 같은 데이터 엘리먼트 ZJMG_ORG_NODE_ID(CHAR10). parent_id가 같은 테이블의 다른 행 키를 가리키는 자기참조가 트리의 전부입니다. 별도 "트리 테이블"은 필요 없습니다.
last_changed_at : timestampl드래프트 전환 때 추가된 필드. BDEF의 total etag가 이 필드를 낙관적 잠금(누가 먼저 고쳤나 감지)의 기준으로 씁니다.

1.2 ZJMG_ORG_NODE_D — 드래프트 테이블

define table zjmg_org_node_d {
  key client    : abap.clnt not null;
  key NodeID    : zjmg_org_node_id not null;
  ParentID      : zjmg_org_node_id;
  OrgName       : zjmg_org_name;
  JobTitle      : zjmg_org_title;
  LastChangedAt : timestampl;
  "~min" : include sych_bdl_draft_admin_inc;
}
줄주해
필드명이 CamelCase드래프트 테이블의 필드명은 CDS 뷰의 요소명과 글자 그대로 일치해야 합니다. 실테이블(snake_case)을 복사해 만들면 "no NODEID field" 오류로 활성화가 거부됩니다(실측).
include sych_bdl_draft_admin_inc드래프트 관리 메타데이터(작성자·시각·처리상태)를 담는 SAP 표준 include. 프레임워크가 자동으로 채웁니다. 편집 중 임시본이 이 테이블에 살다가, 저장(Activate)하면 실테이블로 옮겨집니다.
draftuuid가 없는 이유UUID 늦은 채번(late numbering)을 쓰는 BO에만 필요합니다. 우리처럼 사용자가 키(NodeID)를 직접 주는 비즈니스 키 BO에 넣으면 오히려 활성화가 거부됩니다.
왜 드래프트가 필요했나 — 기술적으론 "저장 전 임시본 + 다중 기기 이어쓰기 + 잠금 완화"가 목적이지만 현실적인 이유가 하나 더 있습니다: Fiori Elements V4 UI 서비스의 List Report는 드래프트가 아니면 표준 Create/Edit 버튼을 그려주지 않습니다. 백엔드가 완벽해도 소용없습니다. V4로 편집 화면을 만든다면 드래프트는 사실상 선택이 아니라 전제입니다.

2부. 루트 뷰 — 모든 것의 단일 원천 기초

@AccessControl.authorizationCheck: #NOT_REQUIRED
@EndUserText.label: 'Org Chart Node (Root)'
define root view entity ZR_JMG_ORG_NODE
  as select from zjmg_org_node
  association [0..1] to ZR_JMG_ORG_NODE as _Parent
    on $projection.ParentID = _Parent.NodeID
{
  key node_id         as NodeID,
      parent_id       as ParentID,
      org_name        as OrgName,
      job_title       as JobTitle,
      last_changed_at as LastChangedAt,
      _Parent
}
줄주해
define root view entityRAP 비즈니스 오브젝트의 최상위 노드임을 선언. Behavior Definition은 root 뷰에만 붙습니다. 주의: 루트는 인터페이스 뷰를 프로젝션해서 만들 수 없어(ROOT 속성 미전파) 이렇게 테이블에서 직접 정의해야 했습니다.
association [0..1] to … as _Parent자기 자신과의 조인 선언. [0..1] = 부모가 없거나(루트 노드) 하나. $projection.ParentID는 "이 뷰가 노출하는 이름 기준"이라는 뜻 — 별칭(as) 이후의 이름으로 조인 조건을 씁니다. 이 _Parent가 뒤의 CDS Hierarchy가 부모를 찾아 올라가는 통로입니다.
node_id as NodeIDDB의 snake_case를 CamelCase로 바꿔 노출. 보기 좋으라고만이 아닙니다 — 이 별칭 체계가 드래프트 테이블 필드명, OData 프로퍼티명, mapping 절 전부의 기준이 됩니다.

3부. Behavior Definition — 행동의 계약서 기초

RAP의 심장입니다. "이 오브젝트로 무엇을 할 수 있고, 그때 무엇이 검증되는가"를 선언형으로 적습니다.

3.1 ZR_JMG_ORG_NODE (BDEF) — 최종본

managed implementation in class zbp_r_jmg_org_node unique;
strict ( 2 );
with draft;

define behavior for ZR_JMG_ORG_NODE
persistent table zjmg_org_node
draft table zjmg_org_node_d
lock master
total etag LastChangedAt
authorization master ( instance )
{
  create ( authorization : global );
  update;
  delete;

  draft action Activate;
  draft action Discard;
  draft action Edit;
  draft action Resume;
  draft determine action Prepare;

  field ( readonly : update, mandatory : create ) NodeID;
  field ( mandatory ) OrgName;

  association _Parent;

  validation checkSelfParent  on save { field ParentID; }
  validation checkParentExists on save { field ParentID; }

  mapping for zjmg_org_node
  {
    NodeID        = node_id;
    ParentID      = parent_id;
    OrgName       = org_name;
    JobTitle      = job_title;
    LastChangedAt = last_changed_at;
  }
}
줄주해
managedINSERT/UPDATE/DELETE·잠금·드래프트 등 기술 코드를 프레임워크가 대신 구현합니다. 우리는 비즈니스 규칙(검증·액션)만 씁니다. 반대 개념인 unmanaged는 저장 로직까지 직접 짭니다(레거시 래핑용).
strict ( 2 ); with draft;strict(2)=최신 문법·베스트프랙티스 강제(클라우드 필수). 선언 순서가 정해져 있습니다: unique; → strict(2); → with draft; — 순서를 바꾸면 활성화 거부(실측).
persistent / draft table실본과 임시본의 저장 위치. 흐름: 편집 → draft 테이블 → 저장(Activate) → 검증 통과 시 persistent 테이블.
lock master / total etag이 엔티티가 잠금의 주인이고, LastChangedAt으로 동시 수정 충돌을 감지합니다(내가 읽은 뒤 남이 고쳤으면 저장 거부).
authorization master ( instance )행 단위 권한 검사 선언. 선언하면 핸들러 구현이 의무 — 미구현이면 조회만 해도 런타임 덤프. create는 행이 아직 없으므로 ( authorization : global )로 전역 검사로 분리.
draft action ×5드래프트 생명주기 표준 5종(Activate/Discard/Edit/Resume/Prepare). 구현은 프레임워크 제공(Prepare 포함 — 핸들러 메서드 불필요, 실측), 선언은 strict 모드라 명시적으로 해야 합니다.
field ( readonly : update, mandatory : create ) NodeID;동작별 필드 제어의 정석: 키는 생성 때는 필수 입력, 수정 때는 잠금. 처음에 무조건 readonly로 했다가 생성 화면에서 키를 못 넣는 문제를 겪고 이렇게 정련했습니다.
validation … on save { field ParentID; }저장 시점 검증 선언. 중괄호 안의 field는 트리거 — ParentID가 바뀐 인스턴스에 대해서만 검증이 돕니다(성능 배려). 구현은 클래스에, 선언은 여기에: 계약과 구현의 분리입니다.
mapping for …CDS 요소명↔DB 컬럼명 대응표. 이름이 다르면 필수 — 없으면 오류 없이 모든 필드가 빈 채 저장되는, 이번 여정에서 가장 음험했던 버그의 주인공.

3.2 ZC_JMG_ORG_MAINT (프로젝션 BDEF) — 화면에 여는 문

projection;
strict ( 2 );
use draft;

define behavior for ZC_JMG_ORG_MAINT
{
  use create;
  use update;
  use delete;

  use action Activate;
  use action Discard;
  use action Edit;
  use action Prepare;
  use action Resume;
}
줄주해
projection / use …루트가 가진 능력 중 이 화면에 노출할 것만 골라 여는 필터입니다. 조회 전용 트리 뷰의 BDEF는 use association _Parent;만 있고 create/update/delete가 없습니다 — 같은 루트, 다른 문. "루트는 전능, 프로젝션이 절제"가 RAP식 읽기전용/편집 분리입니다.
use draft; + use action …드래프트 능력도 명시적으로 열어야 합니다. 이게 빠지면 루트가 draft여도 화면에 Create 버튼이 안 뜹니다(실측 — Create 버튼 미스터리의 마지막 조각이었습니다).
validation이 안 보이는 이유use validation이라는 문법은 존재하지 않습니다. 검증은 create/update가 열려 있으면 루트에서 자동으로 따라옵니다.

4부. 행동 구현 클래스 — 규칙이 사는 곳 기초

4.1 전역 클래스는 껍데기

CLASS zbp_r_jmg_org_node DEFINITION
  PUBLIC ABSTRACT FINAL FOR BEHAVIOR OF ZR_JMG_ORG_NODE.
ENDCLASS.
CLASS zbp_r_jmg_org_node IMPLEMENTATION.
ENDCLASS.

실제 로직은 전역 클래스가 아니라 Local Types(구현 인클루드)의 로컬 핸들러 클래스에 삽니다. FOR BEHAVIOR OF가 이 클래스를 BDEF와 묶고, 로컬 클래스는 cl_abap_behavior_handler를 상속합니다.

4.2 검증 구현 — 자기참조 금지

CLASS lhc_ZR_JMG_ORG_NODE DEFINITION INHERITING FROM cl_abap_behavior_handler.
  PRIVATE SECTION.
    METHODS checkSelfParent FOR VALIDATE ON SAVE
      IMPORTING keys FOR ZR_JMG_ORG_NODE~checkSelfParent.
    ...
ENDCLASS.

METHOD checkSelfParent.
  READ ENTITIES OF ZR_JMG_ORG_NODE IN LOCAL MODE
    ENTITY ZR_JMG_ORG_NODE
      FIELDS ( NodeID ParentID )
      WITH CORRESPONDING #( keys )
    RESULT DATA(lt_node).

  LOOP AT lt_node INTO DATA(ls_node).
    IF ls_node-NodeID = ls_node-ParentID.
      APPEND VALUE #( %tky = ls_node-%tky ) TO failed-ZR_JMG_ORG_NODE.
      APPEND VALUE #( %tky = ls_node-%tky
                      %msg = NEW_MESSAGE( id       = 'ZJMG_ORG_MSG'
                                          number   = '001'
                                          v1       = ls_node-NodeID
                                          severity = if_abap_behv_message=>severity-error ) )
             TO reported-ZR_JMG_ORG_NODE.
    ENDIF.
  ENDLOOP.
ENDMETHOD.
요소주해
FOR VALIDATE ON SAVE … IMPORTING keys FOR 엔티티~검증명RAP 전용 메서드 시그니처. 파라미터를 내가 정하는 게 아니라 BDEF 선언에서 타입이 파생됩니다. keys = 검증 대상 인스턴스들의 키 테이블.
READ ENTITIES … IN LOCAL MODEEML(Entity Manipulation Language) — RAP BO를 다루는 SQL 같은 내장 언어. IN LOCAL MODE는 "BO 내부 접근이니 권한/기능제어를 건너뛰라"는 뜻으로, 검증 중 데이터 재조회의 정석입니다. 드래프트 검증이므로 이 READ는 드래프트 임시본을 읽습니다 — 직접 SELECT 하면 안 되는 이유.
failed / reported + %tky / %msg보이지 않는 응답 파라미터. failed에 키를 넣으면 그 인스턴스의 저장이 거부되고, reported의 %msg가 화면 메시지가 됩니다. %tky는 "transactional key"(드래프트 여부 포함 완전 키), NEW_MESSAGE는 T100 메시지(ZJMG_ORG_MSG 001)를 만드는 RAP 내장 매크로.
checkParentExists (요지)같은 골격에 SELECT SINGLE로 부모 존재를 확인해 없으면 메시지 002. 실전 테스트에서 엑셀 셀 앞 공백이 만든 " MGR001"을 정확히 잡아낸 바로 그 검증입니다.
get_global/get_instance_authorizations권한 핸들러(데모라 all-allow 스텁). requested_authorizations-%create = if_abap_behv=>mk-on으로 "무엇을 물어보는지" 확인하고 auth-allowed를 돌려줍니다. 스텁이라도 구현 자체는 의무 — 없으면 READ에서 덤프.

5부. CDS Hierarchy — 트리를 선언하다 심화

5.1 ZI_JMG_ORG_HIER

@EndUserText.label: 'Org Chart Hierarchy'
define hierarchy ZI_JMG_ORG_HIER
  as parent child hierarchy(
    source
      ZR_JMG_ORG_NODE
      child to parent association _Parent
      start where
        ParentID is initial
      siblings order by
        OrgName ascending
      multiple parents not allowed
  )
  {
    key NodeID,
        ParentID,
        OrgName,
        JobTitle
  }
절주해
define hierarchy재귀 CTE를 손으로 짜는 대신, HANA 계층 엔진에게 "이 뷰를 이 관계로 트리로 펼쳐라"라고 선언합니다. 별도 오브젝트 타입이 아니라 DDLS의 한 종류입니다.
source ZR_JMG_ORG_NODE트리를 펼칠 원천 뷰. 소비뷰가 프로젝션하는 계보와 반드시 같은 뷰여야 합니다 — 처음에 다른 뷰를 source로 썼다가 활성화는 통과하고 Preview에서만 오류가 났습니다.
child to parent association _Parent"자식에서 부모로 가는 길"로 자기참조 association을 지정. 엔진은 이 길을 거꾸로 타고 내려가며 트리를 만듭니다.
start where ParentID is initial루트 노드의 정의 — 부모가 비어 있는 행(CEO)에서 시작.
siblings order by / multiple parents not allowed형제 정렬 기준과 "한 노드에 부모 하나" 제약. 문법 주의: not allowed이지 allowed false가 아닙니다.
본문의 key NodeIDSAP 예제 중엔 key 없는 것도 있으나, 이 시스템에선 소비뷰와 키 구조가 일치해야 Preview가 뜹니다(둘째 Preview 전용 오류의 원인).

5.2 ZC_JMG_ORG_TREE — 트리를 화면에 연결

@AccessControl.authorizationCheck: #NOT_REQUIRED
@EndUserText.label: 'Org Chart Tree'
@OData.hierarchy.recursiveHierarchy: [{entity.name: 'ZI_JMG_ORG_HIER'}]
define root view entity ZC_JMG_ORG_TREE
  provider contract transactional_query
  as projection on ZR_JMG_ORG_NODE
{
  key NodeID,
      ParentID,
      @UI.lineItem: [{ position: 10, label: 'Name' }]
      OrgName,
      @UI.lineItem: [{ position: 20, label: 'Job Title' }]
      JobTitle,
      _Parent : redirected to ZC_JMG_ORG_TREE
}
줄주해
@OData.hierarchy.recursiveHierarchy이 한 줄이 "이 엔티티는 저 Hierarchy 정의대로 트리다"를 OData 메타데이터에 싣습니다. Fiori Elements가 이를 읽어 트리 테이블로 렌더링(2311+ 클라우드 릴리즈). Hierarchy 자체를 서비스에 노출할 필요는 없습니다.
provider contract transactional_query"RAP 트랜잭션 엔진을 통해 조회하라"는 계약이자 위 어노테이션의 전제 조건 — 조회 전용인데도 루트+BDEF 스택이 필요했던 이유입니다.
_Parent : redirected to …루트끼리 연결된 association을 "이 소비뷰 세계 안"으로 다시 겨냥합니다. 프로젝션이 association을 노출할 때의 관용구.

6부. UI 어노테이션 — 화면은 선언으로 그린다 기초

CRUD 화면(ZC_JMG_ORG_MAINT)에는 UI5 코드가 한 줄도 없습니다. 화면 배치 전체가 CDS 어노테이션입니다.

define root view entity ZC_JMG_ORG_MAINT
  provider contract transactional_query
  as projection on ZR_JMG_ORG_NODE
{
      @UI.facet: [{ id: 'GeneralFacet',
                    type: #FIELDGROUP_REFERENCE,
                    label: 'General Information',
                    targetQualifier: 'General',
                    position: 10 }]
      @UI.lineItem:   [{ position: 5,  label: 'Node ID' }]
      @UI.fieldGroup: [{ qualifier: 'General', position: 5, label: 'Node ID' }]
  key NodeID,

      @Consumption.valueHelpDefinition: [{
        entity: { name: 'ZI_JMG_ORG_VH', element: 'NodeID' } }]
      @UI.lineItem:   [{ position: 10, label: 'Parent' }]
      @UI.fieldGroup: [{ qualifier: 'General', position: 20, label: 'Parent' }]
      ParentID,

      @UI.lineItem:   [{ position: 20, label: 'Name' }]
      @UI.fieldGroup: [{ qualifier: 'General', position: 10, label: 'Name' }]
      @UI.selectionField: [{ position: 10 }]
      @Search.defaultSearchElement: true
      OrgName,
      ...
}
어노테이션주해
@UI.lineItemList Report(목록)의 컬럼. position이 순서. 빼먹으면 그 필드는 목록에서 사라집니다 — 실제로 처음엔 NodeID를 빼먹어 "내 ID가 안 보인다"는 화면이 됐습니다.
@UI.fieldGroup + @UI.facetfieldGroup은 "필드 묶음"(qualifier가 묶음 이름), facet은 그 묶음을 Object Page의 섹션으로 배치하는 선언. facet 없이 fieldGroup만 있으면 상세화면이 텅 빕니다(실측). 위치 규칙: 이 시스템에선 @UI.facet을 뷰 헤더나 DDLX가 아니라 키 필드의 첫 인라인 어노테이션으로 두어야 통과합니다.
@Consumption.valueHelpDefinitionF4 값도움말. 전용 뷰 ZI_JMG_ORG_VH의 NodeID 목록에서 고르게 합니다 — 존재하지 않는 부모 입력을 UI 단계에서부터 어렵게 만드는 1차 방어선(2차가 checkParentExists 검증).
@UI.selectionField / @Search.defaultSearchElement필터바의 필터 필드, 상단 검색창의 검색 대상 지정.
팝업이냐 드롭다운이냐 — 값도움말 뷰(ZI_JMG_ORG_VH)의 첫 버전에는 @ObjectModel.resultSet.sizeCategory: #XS가 있었는데, 이러면 Fiori가 "작은 목록"으로 판단해 드롭다운 리스트박스로 렌더링합니다. 이 줄을 제거하면 검색·테이블이 있는 표준 값도움말 다이얼로그(팝업)가 됩니다. 또 드래프트 엔티티 자신을 값도움말 소스로 쓰면 F4가 아예 안 뜨는 경우가 있어, 가볍고 드래프트 없는 전용 VH 뷰를 분리하는 것이 안전합니다.

7.1 파일이 사는 뷰 — largeObject

요소주해
@Semantics.largeObject이 필드(rawstring)가 "파일"임을 선언 — MIME 타입·파일명은 저 필드들이 담당한다는 참조 포함. 이 어노테이션 하나로 Object Page에 파일 업로더 컨트롤이 표준 렌더링됩니다. Fiori Elements에서 파일을 받는 유일한 표준 통로입니다.
UploadID + numbering : managed (BDEF)키는 sysuuid_x16이고 BDEF에서 field ( readonly, numbering : managed ) UploadID; — 사용자가 키를 넣지 않고 프레임워크가 UUID를 자동 채번합니다. 문서·전표류 BO의 표준 채번 방식.
action processUpload result [1] $self; (BDEF)인스턴스 액션 선언. 화면에는 @UI.lineItem/identification의 type: #FOR_ACTION으로 버튼이 됩니다. result [1] $self는 "실행 후 자기 자신을 돌려준다"(화면 갱신용).

7.2 XCO XLSX 파싱 파이프라인

xco_cp_xlsx=>document->for_file_content( ls_upload-Attachment
  )->read_access( )->get_workbook(
  )->worksheet->at_position( 1
  )->select( xco_cp_xlsx_selection=>pattern_builder->simple_from_to(
      )->from_column( xco_cp_xlsx=>coordinate->for_alphabetic_value( 'A' )
      )->to_column(   xco_cp_xlsx=>coordinate->for_alphabetic_value( 'D' )
      )->from_row(    xco_cp_xlsx=>coordinate->for_numeric_value( 2 )
      )->get_pattern( )

  )->if_xco_xlsx_ra_operation~execute( ).
요소주해
XCO_CP_XLSXABAP Cloud에서 릴리즈된 표준 엑셀 라이브러리(XCO = Extension Components). 온프렘의 OLE/ALSM_EXCEL류는 클라우드에 없습니다 — 이게 그 자리의 공식 후계자입니다.
파이프라인 읽는 법문서 열기 → 읽기 접근 → 워크북 → 1번 시트 → A~D열, 2행부터라는 선택 패턴 → 행 스트림 → 내부 테이블(lt_rows)에 부어넣기 → 실행. 컬럼 순서가 구조체 컴포넌트 순서(node_id, parent_id, org_name, job_title)에 그대로 매핑됩니다. 1행(헤더)은 from_row(2)로 자연스럽게 건너뜁니다.
CONDENSE 방어선파싱 직후 ID 필드는 CONDENSE … NO-GAPS(모든 공백 제거), 이름류는 CONDENSE(앞뒤 정리). 실전 엑셀엔 잡공백이 반드시 있고, " MGR001"은 존재하지 않는 부모입니다 — 실제로 겪고 넣은 코드입니다.

7.3 cross-BO EML — 검증을 우회하지 않는 대량 생성

MODIFY ENTITIES OF ZR_JMG_ORG_NODE
  ENTITY ZR_JMG_ORG_NODE
    CREATE AUTO FILL CID
    FIELDS ( NodeID ParentID OrgName JobTitle )
    WITH VALUE #( FOR ls_row IN lt_rows
                   ( NodeID   = ls_row-node_id
                     ParentID = ls_row-parent_id
                     OrgName  = ls_row-org_name
                     JobTitle = ls_row-job_title ) )
  MAPPED   DATA(ls_mapped)
  FAILED   DATA(ls_failed)
  REPORTED DATA(ls_reported).
요소주해
왜 INSERT가 아니라 EML인가업로드 액션 안에서 INSERT zjmg_org_node를 직접 날리면 빠르지만, 노드 BO의 모든 검증·잠금·드래프트 일관성을 전부 우회합니다. EML CREATE는 다른 BO(노드)의 정식 정문으로 들어가므로 자기참조 금지·부모 존재 검증이 업로드 데이터에도 똑같이 작동합니다 — 이 문서 전체에서 가장 강조하고 싶은 한 줄입니다.
CREATE AUTO FILL CIDCID는 생성 요청의 임시 식별자. 결과를 개별 추적할 필요가 없으면 AUTO FILL로 맡깁니다.
FAILED 확인ls_failed가 비지 않으면(중복 키·검증 실패 등) 업로드 인스턴스의 failed/reported에 실어 화면에 오류로 띄웁니다. 성공 시엔 "N nodes created from file" 성공 메시지.

7.4 서비스 — 한 서비스에 두 엔티티

define service ZUI_JMG_ORG_MAINT_O4 {
  expose ZC_JMG_ORG_MAINT as OrgMaint;
  expose ZC_JMG_ORG_UPLD  as OrgUpload;
}

Service Definition은 "무엇을 내보낼까", Service Binding은 "어떤 프로토콜로"(여기선 OData V4 – UI) 담당. 바인딩은 활성화 후 게시(Publish)까지 해야 URL이 열립니다. Eclipse에서 바인딩 더블클릭 → 엔티티 선택 → Preview로 화면을 띄웁니다.

이 편의 한 문장 — "선언은 BDEF에, 규칙은 핸들러에, 화면은 어노테이션에." RAP의 모든 오브젝트는 이 한 문장의 변주입니다. 3부에서 이 구성이 얼마나 정석인지, 실무는 무엇을 더 요구하는지 따집니다.
