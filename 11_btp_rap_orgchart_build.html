1부. BTP Trial ABAP Environment — 누구나 쓰는 무료 클라우드 ABAP 기초

SAP BTP Trial은 SAP가 무료로 제공하는 클라우드 플랫폼 체험 계정입니다. account.hanatrial.ondemand.com에서 누구나 가입할 수 있고, 그 안의 ABAP Environment(코드명 "Steampunk")가 이 문서의 무대입니다. 부스터(Booster)라는 자동화 마법사를 한 번 실행하면 서브계정·인스턴스가 만들어지고, Eclipse의 ADT(ABAP Development Tools)로 접속해 바로 개발을 시작할 수 있습니다.

1.1 온프렘과 결정적으로 다른 점

항목내용
SAP GUI가 없다SE80·SE38·SE11 같은 트랜잭션 화면 자체가 존재하지 않습니다. 개발은 전부 ADT, 실행 화면은 전부 Fiori입니다.
클래식 화면 기술이 없다REPORT, CALL SCREEN, ALV Grid는 문법 차원에서 막혀 있습니다. 화면이 있는 앱은 RAP → OData → Fiori Elements 경로가 사실상 유일합니다.
ABAP Cloud 언어 버전 강제릴리즈된(released) 표준 API만 쓸 수 있고, 표준 테이블 직접 SELECT 등이 활성화 단계에서 차단됩니다. 문법 자체(루프·클래스·CDS)는 아는 ABAP 그대로 — "무엇을 참조할 수 있는가"만 다릅니다.
공유 인스턴스전 세계 trial 사용자가 한 시스템을 씁니다. 내 오브젝트가 남에게 보이고 편집도 가능합니다. 고유 접두사(이 프로젝트는 ZJMG_)로 충돌을 피하고, 계정정보·업무데이터는 절대 올리지 않습니다.
최대 90일 후 삭제인스턴스가 소멸합니다. 시스템을 저장소로 믿지 말고 소스를 즉시 로컬 백업해야 합니다. (이 프로젝트 전체 소스는 3부 부록에 전문 수록.)
왜 이 환경이 학습에 좋은가 — 온프렘 개발 시스템은 "봐주는" 것이 많아 낡은 습관이 통합니다. Trial은 S/4HANA Cloud Public Edition과 같은 클린코어 제약(릴리즈 API·언어 버전·Fiori-only UX)을 그대로 강제하므로, 여기서 익힌 감각은 최신 클라우드 프로젝트에 그대로 이식됩니다. 무료이고, 망가뜨려도 90일 뒤 사라지니 마음껏 실험할 수 있습니다.

3부. 만든 순서와 여정 — 3라운드 스토리 기초

아래 순서는 "이론상 순서"가 아니라 실제로 만들면서 확정된 순서입니다. 각 단계의 산출물이 다음 단계의 전제가 됩니다.

3.1 — 1라운드: 조회 (CDS Hierarchy 트리)

① 데이터 엘리먼트 → 테이블 → 더미 데이터

DTEL 3종을 만들고 ZJMG_ORG_NODE 테이블을 정의했습니다. 테이블 DDL에는 @AbapCatalog.enhancement.category가 사실상 필수 — 빼면 체크는 통과하는데 저장이 거부되는 함정이 있습니다. 데이터는 if_oo_adt_classrun 클래스 (ZCL_JMG_ORG_DATA_LOAD)로 CEO 1명·매니저 2명·팀원 3명을 적재했습니다. GUI가 없으니 SE16으로 넣을 수 없고, 이 "콘솔 클래스" 패턴이 이 환경의 SE38 대용입니다.

② 루트 뷰 + 최소 BDEF

ZR_JMG_ORG_NODE를 테이블에서 직접 정의했습니다(자기참조 association _Parent 포함). 첫 교훈: 루트 뷰는 인터페이스 뷰를 프로젝션해서 만들 수 없습니다 (ROOT 속성이 전파되지 않음). 또 트리 어노테이션이 요구하는 provider contract transactional_query는 활성화된 Behavior Definition이 있는 루트를 전제합니다 — 그래서 "조회 전용인데도" BDEF가 필요했습니다.

③ CDS Hierarchy 정의

ZI_JMG_ORG_HIER — define hierarchy로 부모·자식 관계를 선언합니다. 문법 함정 둘: multiple parents not allowed(× allowed false)이고, 본문 요소에는 소비뷰와 동일하게 key 표시가 필요했습니다. 이 오브젝트가 바로 온프렘(UI5 1.96)에서는 만들어도 그려줄 렌더러가 없어 보류됐던 지점입니다 — 트리 테이블 렌더링은 2311+ 클라우드 릴리즈부터라 Trial에서만 완주할 수 있었습니다.

④ 소비 뷰 + 서비스 + 게시

ZC_JMG_ORG_TREE에 @OData.hierarchy.recursiveHierarchy로 계층 정의를 연결하고 서비스 정의/바인딩(OData V4 UI)을 만들어 게시했습니다. Hierarchy의 source와 소비뷰의 프로젝션 계보는 같은 뷰여야 합니다 — 어기면 활성화는 통과하는데 Preview에서만 오류가 납니다. (중요 교훈: "활성화 성공 ≠ 소비 가능".) Eclipse에서 서비스 바인딩 더블클릭 → Preview로 트리 렌더링을 확인했습니다.

3.2 — 2라운드: 관리 (CRUD 화면)

① 메시지 클래스 + CRUD 소비뷰

ZJMG_ORG_MSG(오류 문구 2건)와 ZC_JMG_ORG_MAINT(List Report/Object Page용 UI 어노테이션 세트)를 생성. @UI.facet은 이 시스템에서 키 필드의 첫 인라인 어노테이션 위치만 허용됩니다 — DDLX 메타데이터 익스텐션은 wrong-scope로 거부(온프렘과 동일 실측).

② 검증 로직 — validation

루트 BDEF에 validation checkSelfParent(자기 자신을 부모로 금지)를 선언하고 구현 클래스의 로컬 핸들러(lhc_)에 구현했습니다. 이후 checkParentExists (존재하지 않는 부모 금지)와 필수 입력(mandatory : create NodeID, mandatory OrgName)을 추가했습니다.

③ Create 버튼이 없다 → 드래프트 전환

가장 큰 반전. 백엔드 CRUD가 완벽해도 OData V4 UI + List Report 조합에서 비(非)드래프트면 표준 Create 버튼 자체가 안 뜹니다. 결국 드래프트 테이블(ZJMG_ORG_NODE_D)을 만들고 BDEF를 with draft로 전환했습니다. 드래프트 테이블의 필드명은 CDS 요소명(CamelCase)과 동일해야 하며, 프로젝션 BDEF에도 use draft;가 있어야 버튼이 나타납니다.

④ 저장했는데 빈 행 → mapping for

첫 생성 테스트에서 값이 전부 빠진 빈 행이 들어갔습니다. 원인은 managed BO에서 CDS 요소명(NodeID)≠DB 컬럼명(node_id)일 때 mapping for 절이 필수라는 것 — 없으면 오류 없이 조용히 빈 값이 저장됩니다. 가장 찾기 어려운 종류의 버그입니다.

⑤ READ에서 덤프 → 권한 핸들러 구현

authorization master (instance)를 선언만 하고 핸들러를 안 만들면 조회만 해도 런타임 덤프(Handler not implemented: INSTANCE_AUTHORIZATION)가 납니다. 활성화 때는 경고(W)로만 지나가서 무시하기 쉽습니다. all-allow 스텁으로라도 get_global/get_instance_authorizations를 구현해 해결했습니다.

⑥ 값도움말 다듬기

ParentID에 F4를 붙이되, 전용 값도움말 뷰 ZI_JMG_ORG_VH를 분리해 연결했습니다 (드래프트 엔티티 자신을 VH 소스로 쓰면 안 뜨는 경우가 있음). 드롭다운이 아니라 팝업 다이얼로그로 나오게 하려면 @ObjectModel.resultSet.sizeCategory: #XS를 빼야 합니다 — 이 어노테이션이 "작은 목록"으로 선언해 드롭다운으로 최적화해버립니다.

3.3 — 3라운드: 대량입력 (엑셀 업로드)

① 충격적 사실: FE에 표준 업로드 버튼이 없다

Excel Export는 표준이지만 Import는 Fiori Elements 표준 기능이 아예 아닙니다 (버전 무관). 그래서 SAP 공식 블로그가 안내하는 패턴을 그대로 채택했습니다: 파일 첨부 필드를 가진 업로드 전용 BO를 만들고 백엔드에서 파싱합니다.

② 업로드 BO — largeObject

ZJMG_ORG_UPLD(+드래프트) 테이블과 ZR/ZC_JMG_ORG_UPLD 스택. 핵심은 @Semantics.largeObject — 이 어노테이션이 붙은 필드는 Object Page에 파일 업로더 컨트롤로 자동 렌더링됩니다. 키는 numbering : managed로 UUID 자동 채번.

③ processUpload 액션 — XCO 파싱 + cross-BO EML

사용자가 파일을 저장한 뒤 "Process Upload" 버튼(액션)을 누르면, XCO_CP_XLSX 모듈로 엑셀을 읽어(A~D열, 2행부터) EML로 노드 BO에 CREATE합니다. DB에 직접 INSERT하지 않고 EML을 타는 것이 핵심 — 자기참조 금지·부모 존재 검증이 업로드 경로에도 그대로 작동합니다. 실전 테스트에서 셀 앞 공백 때문에 "부모 없음" 검증에 걸리는 실제 상황까지 겪었고, 파싱 직후 CONDENSE로 공백을 정리하는 방어 코드를 넣었습니다.
