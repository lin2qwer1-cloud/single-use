1부. 왜 AMDP인가

1.1 Code Pushdown 복습 기본

계산을 데이터가 있는 DB로 내리는 원칙은 CDS 개념 가이드 1부에서 다뤘습니다. AMDP도 같은 목적이지만, 순수 SELECT로 표현하기 어려운 로직(다단계 계산, 절차적 흐름, 복잡한 분기)을 DB에서 실행하기 위한 수단입니다.

1.2 CDS vs AMDP — 무엇을 언제 기본

요구를 하나의 SELECT(조인·집계·CASE·association)로 표현 가능?
        │
   ┌────┴─────┐
  예           아니오
   │             │
 CDS View    다단계 계산·중간결과 재사용·절차 로직·복잡한 SQLScript 필요?
                 │
            ┌────┴────┐
           예           아니오
            │             │
          AMDP     CDS Table Function(간단) 또는 CDS로 재검토

원칙: 선언형으로 표현되면 CDS, 안 되면 AMDP. AMDP를 쓰더라도 내부는 최대한 선언형(SELECT 체이닝)으로 작성합니다(→ 6부 품질 체크리스트).

1.3 AMDP란 기본

AMDP(ABAP Managed Database Procedure)는 ABAP 클래스 메서드로 관리되는 HANA 데이터베이스 프로시저/함수입니다. 메서드 본문을 SQLScript로 작성하면, 활성화 시 SAP가 HANA에 실제 DB 프로시저를 생성·배포합니다.

ABAP 클래스(AMDP 메서드, SQLScript 본문)
        │ 활성화
        ▼
HANA에 DB 프로시저/함수 배포  ── ABAP에서 일반 메서드처럼 호출

클래식한 native SQL(EXEC SQL) 대비 장점: ABAP 수명주기·전송(CTS)으로 관리, 문법 검사, where-used, 디버깅 지원.

2부. AMDP 기본 구조

2.1 클래스 요건 & 메서드 선언 기본

CLASS zcl_x DEFINITION PUBLIC.
  PUBLIC SECTION.
    INTERFACES if_amdp_marker_hdb.        " AMDP 마커 인터페이스(필수)
    CLASS-METHODS get_data
      IMPORTING VALUE(iv_carrid)  TYPE s_carr_id
      EXPORTING VALUE(et_flights) TYPE tt_flight.
ENDCLASS.
CLASS zcl_x IMPLEMENTATION.
  METHOD get_data
      BY DATABASE PROCEDURE FOR HDB LANGUAGE SQLSCRIPT
      OPTIONS READ-ONLY USING sflight.
    et_flights = SELECT ... FROM sflight WHERE carrid = :iv_carrid;
  ENDMETHOD.
ENDCLASS.
  • if_amdp_marker_hdb를 구현해야 AMDP 클래스로 인식됩니다.
  • 메서드 본문은 BY DATABASE PROCEDURE FOR HDB LANGUAGE SQLSCRIPT로 시작.
  • SQLScript 안에서 ABAP 파라미터는 :iv_carrid처럼 콜론으로 참조.

2.2 파라미터 규칙 기본

규칙내용
값 전달만모든 파라미터는 VALUE(...). 참조 전달 불가
타입 제약스칼라(ABAP 사전 타입) 또는 테이블 타입. 중첩 구조/참조 불가
RETURNING 불가EXPORTING/CHANGING만. 결과 테이블은 EXPORTING
인라인 타입 불가메서드 시그니처에 TYPE p LENGTH n DECIMALS n/TYPE c LENGTH n 직접 불가 → 클래스 TYPES로 별칭 선언 후 사용
실전 함정 — IMPORTING VALUE(iv_amt) TYPE p LENGTH 9 DECIMALS 2는 문법 오류. TYPES ty_amount TYPE p LENGTH 9 DECIMALS 2.를 만들고 TYPE ty_amount로 쓰세요. 예제 A1 ▸

2.3 USING · OPTIONS 기본

  • USING sflight scarr — SQLScript가 참조하는 테이블/뷰/다른 AMDP를 명시(실제 참조만).
  • OPTIONS READ-ONLY — 읽기 전용 보장(부수효과 차단 + 최적화 힌트). 기본으로 사용 권장.

2.4 ABAP에서 호출 & 예외 기본

일반 메서드처럼 호출합니다. DB 실행 중 오류는 CX_AMDP_EXECUTION_FAILED(및 관련 cx_amdp_*)로 올라옵니다.

TRY.
    zcl_x=>get_data( EXPORTING iv_carrid = 'AA' IMPORTING et_flights = DATA(lt) ).
  CATCH cx_amdp_execution_failed INTO DATA(lx).
    ...
ENDTRY.

3부. SQLScript 핵심

3.1 테이블 변수 & 선언형 체이닝 기본

lt_flights = SELECT mandt, carrid, connid, paymentsum
               FROM sflight
              WHERE mandt = SESSION_CONTEXT('CLIENT')
                AND seatsocc >= :iv_min_occ;          -- 1단계: 필터 조기
lt_joined  = SELECT f.carrid, c.carrname, f.paymentsum
               FROM :lt_flights AS f                  -- :lt_flights 재사용
               INNER JOIN scarr AS c ON c.carrid = f.carrid;
et_chain   = SELECT carrid, carrname, SUM( paymentsum ) AS totalpay
               FROM :lt_joined GROUP BY carrid, carrname; -- 3단계: 집계는 마지막

중간 결과를 테이블 변수에 담고 :변수로 다음 단계에 넘깁니다. 엔진이 전체 체인을 한 번에 최적화하므로, 사람이 읽기 좋으면서 성능도 좋습니다. 예제 A2-1 ▸

3.3 윈도우 함수 심화

그룹으로 나눈 뒤(PARTITION BY) 각 행에 순위·누적·이전값 등을 계산합니다. GROUP BY와 달리 행이 사라지지 않습니다.

-- 항공사별 매출 순위 (그룹별 Top-N 패턴)
RANK() OVER( PARTITION BY carrid ORDER BY paymentsum DESC ) AS rnk
-- 누적합 / 이전 행 값
SUM( paymentsum ) OVER( PARTITION BY carrid ORDER BY fldate ) AS cumsum
LAG( paymentsum ) OVER( PARTITION BY carrid ORDER BY fldate ) AS prevpay
함수용도
RANK() / DENSE_RANK() / ROW_NUMBER()순위(동점 처리 방식 차이)
SUM/AVG ... OVER(ORDER BY)누적합/이동 집계
LAG() / LEAD()이전/다음 행 값

"항공사별 Top-3"는 rnk를 구한 뒤 바깥에서 WHERE rnk <= 3로 거릅니다. 예제 A2-3/A2-4 ▸

3.4 스칼라 변수 · IF 심화

DECLARE lv_mode NVARCHAR(3) := :iv_mode;
IF :lv_mode = 'AVG' THEN
  et = SELECT carrid, AVG( ... ) AS metric FROM ... GROUP BY carrid;
ELSE
  et = SELECT carrid, SUM( ... ) AS metric FROM ... GROUP BY carrid;
END IF;

DECLARE로 선언, :=로 대입, :변수로 참조. 분기는 되도록 SELECT 개수를 늘리기보다 조건을 SQL 안으로 넣는 편이 낫습니다. 예제 A2-5 ▸

4부. 명령형 & 예외

4.1 커서/LOOP — 문법은 알되, 피하라 심화

SQLScript도 커서와 FOR ... DO 루프를 지원하지만, 행 단위 처리는 컬럼 엔진의 집합 연산 최적화를 포기하는 것입니다. 같은 결과를 대부분 한 문장의 선언형으로 쓸 수 있습니다.

✗ 나쁜 예 — 커서 루프
DECLARE CURSOR lc FOR
  SELECT carrid, paymentsum FROM sflight ORDER BY carrid;
FOR ls AS lc DO
  -- 행마다 누적/분기...
END FOR;
✓ 좋은 예 — 선언형 한 문장
et_sum = SELECT carrid, SUM( paymentsum ) AS totalpay
           FROM sflight
          GROUP BY carrid;

실습 A3에서 두 방식이 동일한 결과(8행, equal=YES)를 내는 것을 러너로 나란히 확인합니다. 예제 A3-1/A3-2 ▸

4.2 예외 처리 심화

DECLARE EXIT HANDLER FOR SQLEXCEPTION
  BEGIN
    ev_result := -1;
    ev_errmsg := SUBSTRING( ::SQL_ERROR_MESSAGE, 1, 100 );
  END;
  • DECLARE EXIT HANDLER FOR SQLEXCEPTION — SQL 오류 발생 시 핸들러로 진입.
  • ::SQL_ERROR_CODE, ::SQL_ERROR_MESSAGE로 오류 정보 접근.
  • ABAP 측에서는 cx_amdp_execution_failed로 캐치.
실전 함정 — ::SQL_ERROR_MESSAGE는 길 수 있습니다(실습에서 156자). 짧은 char 필드에 그대로 대입하면 대입 자체가 2차 오류를 냅니다. SUBSTRING(...,1,n)으로 잘라 담으세요. 예제 A3-3 ▸

4.3 동적 SQL — 기본은 "쓰지 않기" 심화

EXEC / EXECUTE IMMEDIATE로 문자열 SQL을 실행할 수 있지만, SQL 주입 위험, 최적화 불가, READ-ONLY 제약 충돌 때문에 교육/실무 모두 기본은 회피입니다. 정적 SQLScript로 표현할 방법을 먼저 찾으세요.

5부. 통합 & 운영

5.1 CDS Table Function 연계 심화

AMDP 함수를 CDS Table Function으로 노출하면 Open SQL에서 뷰처럼 쓸 수 있고, CDS 파라미터가 AMDP로 전달됩니다.

// CDS 쪽
define table function ZI_JNC_AMDP_TF_TOPN
  with parameters p_top_n : abap.int4
  returns { client : abap.clnt; airlineid : s_carr_id; ...; rnk : abap.int4; }
  implemented by method ZCL_JNC_AMDP_04_INTEGRATION=>get_top_flights;
-- AMDP 쪽: CDS 파라미터를 :p_top_n 으로
RETURN SELECT ... FROM ( ... RANK() OVER(...) AS rnk ... )
        WHERE rnk <= :p_top_n;
CDS 예제 17 (ZI_JNC_EDU_17_TABFUNC)AMDP 예제 A4-1 (ZI_JNC_AMDP_TF_TOPN)
파라미터없음있음(p_top_n)
로직단순 집계윈도우 함수 + 파라미터 필터

5.2 클라이언트 처리 심화

AMDP는 ABAP처럼 자동으로 클라이언트를 걸어주지 않습니다. 누가 mandt 필터를 거는지를 명확히 해야 합니다.

방식내용
수동 필터SQLScript에서 WHERE mandt = SESSION_CONTEXT('CLIENT') 직접
CDS SESSION CLIENT메서드 선언에 AMDP OPTIONS CDS SESSION CLIENT CURRENT → SESSION_CONTEXT('CDS_CLIENT')에 현재 클라이언트 주입
실측 — 실습 A4-2에서 현재 클라이언트 필터는 870행, 무필터는 886행(타 클라이언트 16행 포함). 필터를 빠뜨리면 다른 클라이언트 데이터가 섞입니다. 예제 A4-2 ▸

5.3 AMDP BAdI · 디버거 심화

  • AMDP BAdI — BAdI 구현을 AMDP로 작성해 DB 레벨 확장 포인트를 제공(개념 소개 수준).
  • AMDP 디버거 — ADT에서 SQLScript 라인에 중단점을 걸고 테이블 변수 내용을 검사할 수 있습니다.

6부. AMDP 품질 체크리스트 ★

AMDP의 품질은 대부분 SQLScript 작성 습관에서 갈립니다. 아래 12항목을 기준으로 리뷰하세요.

① 선언형 우선 기본

커서/LOOP/스칼라 반복 대신 SELECT 체이닝으로. 행 단위 처리는 최후의 수단.

② SELECT * 금지 기본

✗
lt = SELECT * FROM sflight;
✓ 필요한 컬럼만
lt = SELECT carrid, connid, paymentsum FROM sflight;

컬럼 스토어는 접근한 컬럼만 읽습니다. 필요한 컬럼만 골라야 이점을 살립니다.

③ 필터를 일찍(Filter Early) 기본

WHERE/파라미터 조건을 체인의 첫 SELECT에 적용해 중간 결과 크기를 줄이세요.

④ READ-ONLY 기본 기본

조회 프로시저는 OPTIONS READ-ONLY. 부수효과를 막고 최적화 힌트가 됩니다.

⑤ USING 최소·정확 기본

실제 참조하는 테이블/뷰/AMDP만 USING에. 불필요한 항목은 의존성만 늘립니다.

⑥ 클라이언트 명시 처리 심화

mandt 필터 규약을 통일하세요. CDS SESSION CLIENT 권장. 빠뜨리면 타 클라이언트 데이터가 섞입니다(5.2 실측 참조).

⑦ ORDER BY는 소비 시점에만 심화

중간 테이블 변수 정렬은 낭비이고, 테이블 변수의 행 순서는 보장되지 않습니다. 최종 결과에서만 정렬하세요.

⑧ 암묵적 형변환 회피 심화

조인 키/비교 대상의 타입을 일치시키세요. 암묵 변환은 인덱스·성능을 해칠 수 있습니다.

⑨ 동적 SQL 금지 기본 심화

EXEC/EXECUTE IMMEDIATE는 주입·최적화 불가. 정적 SQLScript로 대체.

⑩ 이름 규약 기본

파라미터 iv_/et_/ct_, 테이블 변수 lt_, 스칼라 lv_. ABAP 키워드 대문자·SQLScript 일관 들여쓰기.

⑪ 큰 중간 결과 경계 심화

불필요한 DISTINCT, UNION(중복 제거 비용) 대신 가능한 UNION ALL. 조인 전에 집계로 줄일 수 있으면 줄이세요.

⑫ 테스트 용이성 기본

if_oo_adt_classrun 러너나 단위 테스트로 실측 검증하는 습관. 이 교육자료의 모든 결과도 실제 실행으로 캡처했습니다.
